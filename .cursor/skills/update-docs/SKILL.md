# Обновление документации после фичи

## Назначение

После завершения реализации фичи приводит документацию в соответствие с кодом: spec, план, roadmap, architecture, TODO, «После MVP».

## Алгоритм

1. **Определи контекст фичи:**
   - Spec: `docs/features/{X.Y}-{slug}.md` (открытый файл или указанный пользователем)
   - План: `docs/plans/{X.Y}-{slug}.md` (то же имя, что у spec)

2. **Обнови spec:**
   - Статус → `done`
   - Добавь «Реализовано: YYYY-MM-DD»
   - Раздел «Решения» — добавь принятые в ходе реализации решения
   - Сверь описание с кодом, исправь расхождения

3. **Обнови план:**
   - Отметь выполненные шаги
   - Сверь с фактической реализацией, исправь расхождения

4. **Обнови `docs/roadmap.md`:**
   - Добавь ссылку на описание фичи в соответствующую фазу/этап
   - Формат: `[X.Y](docs/features/{X.Y}-{slug}.md)` или аналогичный

5. **При необходимости — `docs/roadmap.md` (разделы TODO и «После MVP»):**
   - Если фича закрыла задачу из TODO — отметь или удали
   - Если появились новые задачи — добавь в TODO или «После MVP»

6. **При необходимости — `docs/architecture.md`:**
   - Если изменилась архитектура (новые модули, паттерны, зависимости) — обнови соответствующие разделы

## Примеры вызова

- `/update-docs`
- «Обнови документацию после фичи 1.2»
- «Фича завершена, синхронизируй spec и roadmap»
- «Update docs for 1.3-services»
