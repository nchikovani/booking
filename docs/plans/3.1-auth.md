# План реализации: Feature 3.1 — Аутентификация Web Admin (вход и регистрация)

**Спецификация:** [docs/features/3.1-auth.md](../features/3.1-auth.md)  
**Область:** `packages/ui`, `apps/admin` (FSD). Backend и БД **не изменяются**; используется Admin Auth API из Feature 1.1.

**Зависимости:** контракт `/api/v1/admin/auth/*`, CORS и cookie (см. 1.1); `VITE_API_URL` согласован с API.

**FSD (критично):** слой **`shared` не импортирует `entities`**. Любой код, читающий Zustand-сессию (`entities/session`), живёт в **`app`** (композиция) или передаётся в **`shared`** только через **инъекцию зависимостей** (колбэки в фабрику клиента). Импорт `react-router` — только из `app` / layouts, не из `shared/api`.

**Типы OpenAPI:** фактический путь в проекте — как в текущем `apps/admin/src/shared/api/client.ts` (например `schema.d.ts` в `src/`). После `pnpm --filter admin api:generate` выровнять импорт с выбранным файлом типов (`openapi.generated.ts` или `schema.d.ts`).

**Уведомления:** toast через **уже подключённый** `SnackbarProvider` в `App.tsx` (отдельный шаг подключения notistack не требуется).

### Шаг 4: HTTP-клиент — FSD-совместимо: `shared` без `entities`, композиция в `app`

**Проблема:** `shared` не может импортировать `entities/session`. Middleware, читающий store, не размещается в `shared` с прямым импортом store.

#### 4.1 `shared/api` — фабрика и утилиты **без домена**

**Файлы/компоненты:**

- `apps/admin/src/shared/api/auth-paths.ts` (или в существующем модуле) — `isAdminAuthCookiePath(url: string): boolean` для путей, где нужен `credentials: 'include'` (минимум: `login`, `register`, `refresh`, `logout`; **`/me`** — включить при сверке с Feature 1.1, если cookie уходит на тот же origin)
- `apps/admin/src/shared/api/create-client-with-auth.ts` — экспорт **`createClientWithAuth(deps, baseUrl)`**, где `deps` — **чистый контракт**, например:
  - `getAccessToken(): string | null`
  - `setAccessToken(token: string): void`
  - `resetSession(): void`
  - `onSessionExpired(): void` — вызывается при неуспешном refresh / втором 401 после retry; **внутри `shared` не вызывать `navigate`** — только этот колбэк
- Обёртка над `fetch` для openapi-fetch: **кастомная логика** (middleware / `customFetch`). **openapi-fetch не даёт готового retry** — реализовать вручную:
  - для auth-путей — подмешивать `credentials: 'include'`
  - для прочих — при наличии токена заголовок `Authorization: Bearer …`; **`credentials: 'include'` не включать** на обычные эндпоинты, если cookie-сессия только на auth-path (сверка с 1.1)
  - при **401** и URL **не** `/refresh`: **single-flight** `POST /refresh`, затем **ровно один** повтор исходного запроса (флаг «retry уже был» на цепочке); при втором 401 — вызвать `resetSession` + `onSessionExpired`, без цикла
- `apps/admin/src/shared/api/client.ts` — либо реэкспорт «текущего» клиента, либо заглушка до инициализации в `app` (см. 4.2); типы `paths` — из **фактического** файла в репозитории

- [ ] Нет импортов из `entities/*` внутри `shared/api` для auth-middleware

#### 4.2 Слой `app` — композиция клиента и колбэк навигации

**Файлы/компоненты:**

- `apps/admin/src/app/api/init-admin-api-client.ts` или `apps/admin/src/app/providers/AdminApiProvider.tsx` — при монтировании приложения:
  - создать клиент через `createClientWithAuth`, передав `getAccessToken` / `setAccessToken` / `resetSession` из **`entities/session`**
  - **`onSessionExpired`**: `resetSession` (если ещё не сброшено) + **`navigate`** на `ROUTE_LOGIN` с безопасным `returnUrl` — через `useNavigate` внутри провайдера или зарегистрированный колбэк из компонента, имеющего доступ к router
- Обеспечить, чтобы фичи импортировали **готовый** `client` (контекст + `useAdminClient()` из `app`, либо присвоение модуля после `init` — выбрать один паттерн и зафиксировать в коде без циклических импортов)

- [ ] Нигде в `shared` нет импорта `react-router-dom`
- [ ] Единая точка «разлогин из-за refresh» совпадает с поведением spec

### Шаг 5: Bootstrap сессии — SessionProvider, статус `ready`, провал `/me`

**Файлы/компоненты:**

- `apps/admin/src/app/providers/SessionProvider.tsx`
- `apps/admin/src/app/providers/useSessionBootstrap.ts` — состояние bootstrap: минимум **`initializing` | `ready`**, плюс признак «аутентифицирован / гость» после `ready`
- `apps/admin/src/app/App.tsx` — порядок: `QueryClientProvider` → инициализация API-клиента (шаг 4.2) → `SessionProvider` → роутер; **SnackbarProvider** уже есть — использовать для toast из фич

- [ ] **Cold start:** нет access в памяти → `POST /refresh` с `credentials: 'include'`
- [ ] Успех refresh → сохранить access → **`GET /me`**; только после успешного `/me` — состояние «вошёл»
- [ ] **Провал `GET /me` после успешного refresh** (401 и т.д.) — считать сессию невалидной: **`resetSession`**, гость, переход в **`ready`**
- [ ] Неуспех refresh без токена — гость, **`ready`**
- [ ] Пока **`bootstrap !== ready`** — глобальный **loading** (не показывать преждевременно экран логина для защищённой зоны)
- [ ] После login/register (фичи) — тот же порядок: token → `/me` → редирект

## Риски и открытые вопросы

- **Циклические импорты:** решать за счёт фабрики в `shared` + инициализация в `app`; избегать `client.ts`, который импортирует и store, и store, который импортирует client.
- **Повтор запроса после 401:** строго один retry на исходный запрос; второй 401 → `onSessionExpired`.
- **Редирект:** только через колбэк из `app`, не из `shared/api`.
- **Тестовый раннер:** Vitest — выбрать и зафиксировать команду `pnpm --filter admin test`.
- **Актуальность типов API:** `pnpm --filter admin api:generate` и единый импорт `paths`.
- **Несколько вкладок:** MVP — single-flight в пределах вкладки; rotation между вкладками — перелогин (spec §4.2).
- **Документация:** при необходимости добавить в `.cursor/rules/frontend.mdc` исключение для Zustand session token (как в spec 3.1).
