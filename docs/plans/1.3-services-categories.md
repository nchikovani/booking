### Шаг 7: Реализация Gap strategy в ServiceService.reorder

**Файлы/компоненты:** `apps/api/src/modules/service/service.service.ts`, `apps/api/src/modules/service/service.repository.ts`

Полная реализация логики reorder (в Шаге 4 — только сигнатура).

- [x] Логика reorder:
  - `afterServiceId === id` → идемпотентно, вернуть текущую услугу
  - `afterServiceId` отсутствует/null → вставка в начало: position = min(position) - 100 или 0, при необходимости ребаланс
  - `afterServiceId` указан:
    - если afterServiceId — последняя услуга (нет следующей) → position = max(position) + 100
    - иначе → найти позиции after и следующей; если есть зазор — вставить посередине; иначе — ребаланс
- [x] Ребаланс: пересчитать все position с шагом 100 (100, 200, 300, …)
- [x] Проверка: `afterServiceId` должен принадлежать businessId, иначе 404

## Риски и открытые вопросы

1. **Gap strategy, race condition при ребалансе:** при параллельных reorder возможны конфликты. В MVP допустимо; после MVP — транзакции или блокировки (см. roadmap).
2. **Порядок маршрутов NestJS:** `admin/businesses/:businessId/categories` и `admin/businesses/:businessId/services` не конфликтуют с `admin/businesses/:id` — параметры `businessId` и `id` семантически совпадают.
3. **Decimal в DTO:** Prisma `Decimal` в JSON — строка. В DTO `price` при вводе — number, в ответе — string (как в spec: `"price": "1500.00"`).
4. **E2E: создание бизнеса:** Feature 1.2 — бизнес создаётся при GET /me. В E2E: логин → GET /admin/auth/me → получить businessId из ответа.
