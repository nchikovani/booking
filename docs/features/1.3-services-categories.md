## 1. Обзор

CRUD для категорий услуг и услуг в Admin API. Категории и услуги принадлежат бизнесу. Услуги поддерживают сортировку, поиск, фильтрацию по категориям и cursor-based пагинацию. Переупорядочивание услуг реализовано через **Gap strategy** (позиции с «зазорами»).

### 1.1. Область реализации

| Область             | Применимо | Примечание                                      |
| ------------------- | --------- | ----------------------------------------------- |
| Backend (API)       | ☑         | NestJS, Admin API эндпоинты                      |
| Frontend (Admin)    | ☐         | Web Admin — страница «Услуги» (Feature 3.3)      |
| Frontend (Mini App) | ☐         | Публичный каталог услуг (Feature 2.3)          |
| База данных         | ☑         | Prisma, ServiceCategory, Service                |

### 1.2. Контекст

| Сущность      | Назначение                                                                  |
| ------------- | --------------------------------------------------------------------------- |
| **ServiceCategory** | Категория услуг (например: «Стрижка», «Окрашивание»). Принадлежит бизнесу |
| **Service**   | Услуга (название, описание, цена, длительность, категория). Принадлежит бизнесу |
| **Business**  | Бизнес — владелец категорий и услуг                                         |

## 4. Технический дизайн

### 4.1. Модель владения

Категории и услуги принадлежат бизнесу (`businessId`). Доступ контролируется через `requireBusinessMember(adminUserId, businessId)`.

### 4.2. Архитектура модулей

```
apps/api/src/modules/
├── business/                    # Существующий (requireBusinessMember)
├── admin/
│   ├── admin.module.ts          # imports: AdminServiceCategoryModule, AdminServiceModule
│   ├── business/
│   └── service-category/        # Admin-специфичный слой
│       ├── admin-service-category.module.ts
│       └── admin-service-category.controller.ts
│   └── service/
│       ├── admin-service.module.ts
│       └── admin-service.controller.ts
│
└── service-category/            # Shared (ядро)
    ├── service-category.module.ts
    ├── service-category.service.ts
    ├── service-category.repository.ts
    └── dto/
└── service/
    ├── service.module.ts
    ├── service.service.ts
    ├── service.repository.ts
    └── dto/
```

**Альтернатива:** один shared-модуль `service/` с `ServiceCategory` и `Service` как подсущностями.

### 4.3. Gap strategy для порядка услуг

- Поле `position` (integer) — позиция в списке
- Начальные позиции: 100, 200, 300, … (шаг 100)
- При вставке между 100 и 200 присваивается 150
- При исчерпании зазора (например, 100 и 101) — ребаланс: пересчёт позиций с шагом 100 (редко). **Риск:** при параллельных reorder возможны race conditions; см. «После MVP».
- Эндпоинт `PATCH /services/:id/reorder` принимает `{ afterServiceId?: string }` — вставка после указанной услуги; если `afterServiceId` отсутствует — в начало
- Обновляется только одна запись (position), без массового обновления

## 5. Схема базы данных

### 5.1. Таблица `ServiceCategory`

```prisma
model ServiceCategory {
  id          String   @id @default(uuid())
  businessId  String   @map("business_id")
  name        String

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  business    Business @relation(fields: [businessId], references: [id], onDelete: Cascade)
  services    Service[]

  @@unique([businessId, name])
  @@index([businessId])
  @@map("service_categories")
}
```

### 5.2. Таблица `Service`

```prisma
model Service {
  id          String   @id @default(uuid())
  businessId  String   @map("business_id")
  categoryId  String?  @map("category_id")
  name        String
  description String?  @db.Text
  price       Decimal  @db.Decimal(10, 2)

  durationMinutes Int     @map("duration_minutes")  // длительность в минутах
  breakAfterMinutes Int?  @map("break_after_minutes") @default(0)  // технический перерыв после услуги (мин)

  position    Int      @default(0)  // для Gap strategy

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  business    Business        @relation(fields: [businessId], references: [id], onDelete: Cascade)
  category    ServiceCategory? @relation(fields: [categoryId], references: [id], onDelete: SetNull)

  @@index([businessId])
  @@index([businessId, categoryId])
  @@index([businessId, position])
  @@map("services")
}
```

### 5.3. Связь с Business

В модели `Business`:

```prisma
serviceCategories ServiceCategory[]
services         Service[]
```

### 6.1. Список категорий

**GET** `/api/v1/admin/businesses/:businessId/categories`

Возвращает все категории бизнеса без пагинации. Сортировка по `name` (asc).

**Response 200:**
```json
{
  "status": "success",
  "data": [
    {
      "id": "uuid",
      "name": "Стрижка",
      "createdAt": "2024-01-15T10:00:00.000Z"
    }
  ]
}
```

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.2. Создание категории

**POST** `/api/v1/admin/businesses/:businessId/categories`

**Request body:**
```json
{
  "name": "Стрижка"
}
```

**Response 201:**
```json
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Стрижка",
    "createdAt": "2024-01-15T10:00:00.000Z"
  }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND` | `409 CONFLICT` (дубликат name в рамках бизнеса)

### 6.3. Обновление категории

**PATCH** `/api/v1/admin/businesses/:businessId/categories/:id`

**Request body:**
```json
{
  "name": "Стрижка и укладка"
}
```

**Response 200:** `{ "status": "success", "data": { /* обновлённая категория */ } }`

**Errors:** `400 VALIDATION_FAILED` (если body пустой или невалидный) | `401 UNAUTHORIZED` | `404 NOT_FOUND` | `409 CONFLICT`

### 6.4. Удаление категории

**DELETE** `/api/v1/admin/businesses/:businessId/categories/:id`

**Поведение:** `categoryId` у связанных услуг устанавливается в `null` (onDelete: SetNull).

**Response 200:** `{ "status": "success", "data": { "message": "Категория удалена" } }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.5. Список услуг

**GET** `/api/v1/admin/businesses/:businessId/services`

**Query params:**

| Параметр | Тип | Описание |
| -------- | --- | -------- |
| cursor | string | Курсор для следующей страницы (опционально) |
| limit | number | Кол-во записей (по умолчанию 20, макс 100) |
| sort | string | Сортировка: `name` (asc), `-name` (desc), `position` (asc), `-position` (desc). По умолчанию `position` |
| search | string | Поиск по названию ИЛИ описанию (ILIKE) |
| categoryId | string | Фильтр по категории (UUID) |

**Response 200:**
```json
{
  "status": "success",
  "data": {
    "items": [
      {
        "id": "uuid",
        "name": "Мужская стрижка",
        "description": "Классическая стрижка",
        "price": "1500.00",
        "durationMinutes": 60,
        "breakAfterMinutes": 0,
        "categoryId": "uuid",
        "categoryName": "Стрижка",
        "employeeIds": ["uuid1", "uuid2"],
        "position": 100,
        "createdAt": "2024-01-15T10:00:00.000Z"
      }
    ],
    "nextCursor": "eyJpZCI6InV1aWQifQ=="
  }
}
```

`nextCursor` — `null`, если нет следующей страницы. При пустом списке: `{ "items": [], "nextCursor": null }`.

**Errors:** `400 BAD_REQUEST` (INVALID_CURSOR, INVALID_CATEGORY) | `401 UNAUTHORIZED` | `404 NOT_FOUND`

| Сценарий | Поведение |
| -------- | --------- |
| Бизнес не найден / нет доступа | 404 |
| `categoryId` не принадлежит бизнесу | 400 INVALID_CATEGORY |
| `categoryId` не найден в БД | 400 INVALID_CATEGORY |
| Невалидный cursor | 400 INVALID_CURSOR |

### 6.6. Создание услуги

**POST** `/api/v1/admin/businesses/:businessId/services`

**Request body:**
```json
{
  "name": "Мужская стрижка",
  "description": "Классическая стрижка машинкой и ножницами",
  "price": 1500,
  "durationMinutes": 60,
  "breakAfterMinutes": 0,
  "categoryId": "uuid",
  "employeeIds": ["uuid1", "uuid2"]
}
```

**Response 201:** `{ "status": "success", "data": { /* созданная услуга */ } }`

**Поведение:** `position` — последняя в списке (max(position) + 100). Связи Employee ↔ Service сохраняются через общий метод синхронизации (см. Feature 1.4). Все `employeeIds` должны принадлежать тому же бизнесу; иначе 404 NOT_FOUND.

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND` (категория или сотрудник не принадлежит бизнесу или не существует)

### 6.7. Получение услуги

**GET** `/api/v1/admin/businesses/:businessId/services/:id`

**Response 200:** `{ "status": "success", "data": { /* услуга */ } }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.8. Обновление услуги

**PATCH** `/api/v1/admin/businesses/:businessId/services/:id`

Частичное обновление. Все поля опциональны. PATCH с пустым body `{}` возвращает 200 с текущими данными услуги (без изменений).

**Request body:**
```json
{
  "name": "Мужская стрижка",
  "description": "Обновлённое описание",
  "price": 1500,
  "durationMinutes": 60,
  "breakAfterMinutes": 0,
  "categoryId": "uuid",
  "employeeIds": ["uuid1", "uuid2", "uuid3"]
}
```

- `employeeIds` — полная замена связей услуги с сотрудниками. Передача `[]` — удаление всех связей. Используется общий метод синхронизации EmployeeService (Feature 1.4).
- Все `employeeIds` должны принадлежать тому же бизнесу; иначе 404 NOT_FOUND.

**Response 200:** `{ "status": "success", "data": { /* обновлённая услуга с employeeIds */ } }`

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND` (категория или сотрудник не принадлежит бизнесу или не существует)

### 6.9. Удаление услуги

**DELETE** `/api/v1/admin/businesses/:businessId/services/:id`

**Response 200:** `{ "status": "success", "data": { "message": "Услуга удалена" } }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.10. Изменение позиции услуги (Gap strategy)

**PATCH** `/api/v1/admin/businesses/:businessId/services/:id/reorder`

**Request body:**
```json
{
  "afterServiceId": "uuid"
}
```

- `afterServiceId` — вставить после указанной услуги
- `afterServiceId: null` или отсутствует — в начало списка
- `afterServiceId === id` — идемпотентно, 200 без изменений
- `position` — середина между `position` предыдущей и следующей позиции; при отсутствии зазора — ребаланс

**Response 200:** `{ "status": "success", "data": { /* обновлённая услуга с position */ } }`

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND`

| Сценарий | Поведение |
| -------- | --------- |
| `afterServiceId` не принадлежит бизнесу | 404 |
| `afterServiceId` не найден | 404 |
| `afterServiceId === id` | 200 идемпотентно |

## 9. Безопасность и доступ

- **Guard:** AdminAuthGuard — извлечение `adminUserId` из JWT
- **Проверка:** `requireBusinessMember(adminUserId, businessId)` — для всех эндпоинтов
- **При невалидном или истёкшем JWT:** 401 `UNAUTHORIZED`
- **При отсутствии доступа:** 404 `NOT_FOUND`
- **При отсутствии доступа к ресурсу (category/service):** 404 `NOT_FOUND` (не раскрывать существование)
