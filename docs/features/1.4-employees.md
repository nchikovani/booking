## 1. Обзор

CRUD для сотрудников в Admin API. Сотрудники принадлежат бизнесу. Сотрудник связан с услугами (many-to-many) и имеет график работы (Feature 1.5). Сотрудник возвращается вместе с графиком; при создании и редактировании можно задать график (шаблон или индивидуальный). Данные сотрудников используются в Client API (Feature 2.3) для отображения в каталоге и при выборе слота записи.

### 1.1. Область реализации

| Область             | Применимо | Примечание                                      |
| ------------------- | --------- | ----------------------------------------------- |
| Backend (API)       | ☑         | NestJS, Admin API; Employee CRUD + schedule (Feature 1.5); расширение Service API (employeeServices/services в DTO, sync при create/update) |
| Frontend (Admin)    | ☐         | Web Admin — страница «Сотрудники» (Feature 3.4) |
| Frontend (Mini App) | ☐         | Публичный каталог (Feature 2.3)                |
| База данных         | ☑         | Prisma, Employee, EmployeeService, EmployeeSchedule (Feature 1.5) |

### 1.2. Контекст

| Сущность        | Назначение                                                                 |
| --------------- | -------------------------------------------------------------------------- |
| **Employee**    | Сотрудник/мастер — имя, фото, специализация, график работы. Принадлежит бизнесу |
| **EmployeeService** | Связь many-to-many между Employee и Service (мастер ↔ услуги)          |
| **EmployeeSchedule** | График работы сотрудника (Feature 1.5): шаблон или индивидуальные дни |
| **Business**    | Бизнес — владелец сотрудников                                              |
| **Service**     | Услуга — может оказываться несколькими сотрудниками                         |

## 4. Технический дизайн

### 4.1. Модель владения

Сотрудники принадлежат бизнесу (`businessId`). Доступ контролируется через `requireBusinessMember(adminUserId, businessId)`.

### 4.2. Архитектура модулей

```
apps/api/src/modules/
├── business/                    # Существующий (requireBusinessMember)
├── admin/
│   ├── admin.module.ts          # imports: AdminEmployeeModule
│   └── employee/
│       ├── admin-employee.module.ts
│       └── admin-employee.controller.ts
│
└── employee/                    # Shared (ядро)
    ├── employee.module.ts
    ├── employee.service.ts
    ├── employee.repository.ts
    └── dto/
```

**Shared-модуль EmployeeModule** — общая логика для Admin API (Feature 1.4) и Client API (Feature 2.3). Admin-слой — только контроллер с проверкой прав.

### 4.3. Связь Employee ↔ Service

- Таблица `EmployeeService` — явная связь many-to-many с полями `priceOverride`, `durationMinutesOverride`
- **Сотрудник:** при создании/обновлении передаётся массив `services` (serviceId, priceOverride?, durationMinutesOverride?); полная замена связей; дубликаты дедуплицируются
- **Услуга:** при создании/обновлении (Feature 1.3) передаётся массив `employeeServices` (employeeId, priceOverride?, durationMinutesOverride?); полная замена связей; дубликаты дедуплицируются
- Общий метод синхронизации (например, `syncEmployeeServiceLinks`) используется и при сохранении сотрудника, и при сохранении услуги
- Проверка существования и принадлежности `serviceIds`/`employeeIds` бизнесу — в одной транзакции с сохранением
- При наличии хотя бы одного невалидного ID — 404 NOT_FOUND (без раскрытия, какой именно)

### 4.4. График работы (Feature 1.5)

- При GET (список, по id) сотрудник возвращается с полем `schedule`: `null` | `{ type: "template", scheduleTemplateId, scheduleTemplateName, days }` | `{ type: "custom", days }`
- При POST/PATCH можно передать `schedule`: `null` (удалить) | `{ scheduleTemplateId }` (шаблон) | `{ days }` (индивидуальный)
- Либо `scheduleTemplateId`, либо `days` — взаимоисключающие
- При `scheduleTemplateId`: удалить EmployeeScheduleDay и EmployeeScheduleDayBreak, установить scheduleTemplateId
- При `days`: установить scheduleTemplateId = null, полная замена дней
- При `schedule: null`: удалить EmployeeSchedule. При отсутствии поля (только для update): не менять график
- При `schedule: { days: [] }`: допустимо — сотрудник без рабочих дней; EmployeeSchedule с пустым days
- scheduleTemplateId должен принадлежать businessId. Проверка существования и принадлежности — в одной транзакции с сохранением; при ошибке — откат и 404

### 4.5. Хранение фото

- Поле `imagePath` — путь в MinIO, например `employees/{employeeId}/photo.webp`
- StorageService: upload, delete, getPublicUrl (по аналогии с Business)
- Конвертация в WebP при загрузке

## 5. Схема базы данных

### 5.1. Таблица `Employee`

```prisma
model Employee {
  id             String   @id @default(uuid())
  businessId     String   @map("business_id")
  name           String
  specialization String?  @map("specialization")
  imagePath      String?  @map("image_path")
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  business         Business         @relation(fields: [businessId], references: [id], onDelete: Cascade)
  employeeServices EmployeeService[]
  employeeSchedule EmployeeSchedule?

  @@index([businessId])
  @@map("employees")
}
```

### 5.2. Таблица `EmployeeService` (связь many-to-many)

```prisma
model EmployeeService {
  id                    String    @id @default(uuid())
  employeeId            String    @map("employee_id")
  serviceId             String    @map("service_id")
  priceOverride         Decimal?  @map("price_override") @db.Decimal(10, 2)
  durationMinutesOverride Int?     @map("duration_minutes_override")
  createdAt             DateTime  @default(now())

  employee Employee @relation(fields: [employeeId], references: [id], onDelete: Cascade)
  service  Service  @relation(fields: [serviceId], references: [id], onDelete: Cascade)

  @@unique([employeeId, serviceId])
  @@index([employeeId])
  @@index([serviceId])
  @@map("employee_services")
}
```

- `priceOverride` (Decimal?, nullable) — индивидуальная цена; при `null` используется `Service.price`
- `durationMinutesOverride` (Int?, nullable) — индивидуальная длительность; при `null` используется `Service.durationMinutes`

### 5.3. EmployeeSchedule (Feature 1.5)

См. [Feature 1.5](1.5-work-schedule.md), раздел 5. Employee имеет связь `employeeSchedule EmployeeSchedule?`.

### 5.4. Обновление моделей Business и Service

В модели `Business`:
```prisma
employees Employee[]
```

В модели `Service`:
```prisma
employeeServices EmployeeService[]
```

### 6.1. Список сотрудников

**GET** `/api/v1/admin/businesses/:businessId/employees`

Без пагинации. Возвращает полный список сотрудников бизнеса. Для MVP ожидается небольшое число сотрудников на бизнес; при росте — рассмотреть пагинацию.

**Query params:**

| Параметр | Тип | Описание |
| -------- | --- | -------- |
| search | string | Поиск по имени (ILIKE). Пустая строка — игнорировать (без фильтра) |
| sort | string | Сортировка: `name` (asc), `-name` (desc). По умолчанию `name`. Невалидное значение — 400 VALIDATION_FAILED |

**Response 200:**
```json
{
  "status": "success",
  "data": [
    {
      "id": "uuid",
      "name": "Анна Иванова",
      "specialization": "Мастер маникюра",
      "imageUrl": "https://...",
      "services": [
        { "serviceId": "uuid1" },
        { "serviceId": "uuid2", "priceOverride": "2000.00", "durationMinutesOverride": 45 }
      ],
      "schedule": {
        "type": "template",
        "scheduleTemplateId": "uuid",
        "scheduleTemplateName": "Стандартный график",
        "days": [
          { "dayOfWeek": 1, "startTime": "09:00", "endTime": "18:00", "breaks": [{ "startTime": "13:00", "endTime": "14:00" }] }
        ]
      },
      "createdAt": "2024-01-15T10:00:00.000Z"
    }
  ]
}
```

`schedule` может быть `null` (нет графика), `{ type: "template", ... }` или `{ type: "custom", days: [...] }`.

При пустом списке: `{ "status": "success", "data": [] }`.

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

| Сценарий | Поведение |
| -------- | --------- |
| Бизнес не найден / нет доступа | 404 |
| Невалидный businessId (не UUID) | 400 VALIDATION_FAILED |

### 6.2. Создание сотрудника

**POST** `/api/v1/admin/businesses/:businessId/employees`

**Request body:**
```json
{
  "name": "Анна Иванова",
  "specialization": "Мастер маникюра",
  "services": [
    { "serviceId": "uuid1" },
    { "serviceId": "uuid2", "priceOverride": 2000, "durationMinutesOverride": 45 }
  ],
  "schedule": {
    "scheduleTemplateId": "uuid"
  }
}
```

При отсутствии override — используются базовые значения услуги. Валидация: `priceOverride` ≥ 0; `durationMinutesOverride` ∈ [5, 480], кратно 5.

`schedule` (опционально): `null` | `{ scheduleTemplateId }` | `{ days: [...] }`. Либо scheduleTemplateId, либо days. Время кратно 5. См. [Feature 1.5](1.5-work-schedule.md).

**Response 201:**
```json
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Анна Иванова",
    "specialization": "Мастер маникюра",
    "imageUrl": null,
    "services": [
      { "serviceId": "uuid1" },
      { "serviceId": "uuid2", "priceOverride": "2000.00", "durationMinutesOverride": 45 }
    ],
    "schedule": { "type": "template", "scheduleTemplateId": "uuid", "scheduleTemplateName": "...", "days": [...] },
    "createdAt": "2024-01-15T10:00:00.000Z"
  }
}
```

**Errors:** `400 VALIDATION_FAILED` (в т.ч. невалидный UUID scheduleTemplateId, пустой schedule {}, оба поля scheduleTemplateId и days) | `401 UNAUTHORIZED` | `404 NOT_FOUND` (услуга или scheduleTemplateId не принадлежит бизнесу или не существует)

| Сценарий | Поведение |
| -------- | --------- |
| Один из serviceId не принадлежит бизнесу или не найден | 404 NOT_FOUND (без раскрытия деталей) |
| services с дубликатами serviceId | дедупликация перед сохранением |
| services пустой массив | 201 — сотрудник создан без услуг |

### 6.4. Обновление сотрудника

**PATCH** `/api/v1/admin/businesses/:businessId/employees/:id`

Частичное обновление. Все поля опциональны. PATCH с пустым body `{}` возвращает 200 с текущими данными (без изменений).

**Request body:**
```json
{
  "name": "Анна Иванова",
  "specialization": "Мастер маникюра",
  "services": [
    { "serviceId": "uuid1" },
    { "serviceId": "uuid2", "priceOverride": 2000, "durationMinutesOverride": 45 }
  ],
  "schedule": { "days": [{ "dayOfWeek": 1, "startTime": "09:00", "endTime": "18:00", "breaks": [] }] }
}
```

- `services` — полная замена связей при передаче; при отсутствии поля — связи не меняются. Передача `[]` — удаление всех связей. Дубликаты дедуплицируются.
- `specialization: ""` — очистка, сохраняется как `null`.
- `schedule` — при передаче: `null` (удалить график), `{ scheduleTemplateId }` (шаблон), `{ days }` (индивидуальный). При отсутствии поля — не менять. При scheduleTemplateId индивидуальный график удаляется.

**Response 200:** `{ "status": "success", "data": { /* обновлённый сотрудник */ } }`

**Errors:** `400 VALIDATION_FAILED` (в т.ч. невалидный UUID scheduleTemplateId, пустой schedule {}, оба поля scheduleTemplateId и days) | `401 UNAUTHORIZED` | `404 NOT_FOUND` (услуга или scheduleTemplateId не принадлежит бизнесу или не существует)

### 6.5. Удаление сотрудника

**DELETE** `/api/v1/admin/businesses/:businessId/employees/:id`

**Поведение:** Hard delete. Каскадное удаление записей в `EmployeeService` и `EmployeeSchedule` (Feature 1.5). Удаление файла фото из MinIO (если есть). Если файла нет в MinIO — считать успехом (идемпотентность).

**Response 200:** `{ "status": "success", "data": null }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.6. Загрузка фото сотрудника

**POST** `/api/v1/admin/businesses/:businessId/employees/:id/photo`

**Поведение:**
- `multipart/form-data`, поле `file`
- Допустимые форматы: JPEG, PNG, WebP. Проверка MIME на сервере. Недопустимый формат — 400 VALIDATION_FAILED
- Размер: max 8 MB (Multer). Превышение — 400 VALIDATION_FAILED
- Отсутствие или пустой `file` — 400 VALIDATION_FAILED
- Конвертация в WebP (sharp) → загрузка в MinIO
- При замене: удаление старого файла перед сохранением нового («последняя загрузка побеждает»)
- В БД сохраняется путь: `employees/{employeeId}/photo.webp`

**Response 200:**
```json
{
  "status": "success",
  "data": { "url": "{FILE_STORAGE_URL}/employees/{id}/photo.webp" }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.7. Удаление фото сотрудника

**DELETE** `/api/v1/admin/businesses/:businessId/employees/:id/photo`

- Удалить файл из MinIO, обновить поле в БД на `null`
- Если файла нет в MinIO — считать успехом, обновить только БД (идемпотентность)

**Response 200:** `{ "status": "success", "data": null }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

## 9. Безопасность и доступ

- **Guard:** AdminAuthGuard — извлечение `adminUserId` из JWT
- **Проверка:** `requireBusinessMember(adminUserId, businessId)` — для всех эндпоинтов
- **При невалидном или истёкшем JWT:** 401 `UNAUTHORIZED`
- **При отсутствии доступа:** 404 `NOT_FOUND`
- **При отсутствии доступа к ресурсу (employee):** 404 `NOT_FOUND` (не раскрывать существование)

**Env:** `FILE_STORAGE_URL`, `FILE_STORAGE_BUCKET` — по аналогии с Feature 1.2.

## 11. Конфигурация и зависимости

**Env-переменные:**

- `FILE_STORAGE_URL` — базовый URL хранилища без завершающего слэша
- `FILE_STORAGE_BUCKET` — имя bucket (по умолчанию `uploads`)

**Зависимости:**

- **Feature 1.1** — аутентификация AdminUser (AdminAuthGuard)
- **Feature 1.2** — Business, BusinessService.requireBusinessMember
- **Feature 1.3** — Service, ServiceModule. **В объём 1.4 входит:** (1) добавление `employeeServices` в `CreateServiceDto` и `UpdateServiceDto`; (2) вызов метода синхронизации из `ServiceService` при create/update; (3) возврат `employeeServices` в ответах списка и получения услуги
- **Feature 1.5** — EmployeeSchedule, ScheduleTemplate. **В объём 1.5 входит расширение 1.4:** (1) поле `schedule` в CreateEmployeeDto, UpdateEmployeeDto; (2) возврат `schedule` в ответах GET employees; (3) синхронизация графика при create/update
- **StorageModule** — StorageService (upload, delete, WebP)
- **Prisma** — Employee, EmployeeService, EmployeeSchedule (Feature 1.5)

## 12. Связь с другими фичами

- **Feature 1.2** — Business, requireBusinessMember
- **Feature 1.3** — Service: при создании/обновлении услуги сохраняются employeeServices через общий метод синхронизации; при удалении Service — каскад на EmployeeService
- **Feature 1.5** — EmployeeSchedule: график управляется через Employee API; при удалении Employee — каскад на EmployeeSchedule
- **Feature 2.3** — Client API: публичный список сотрудников (чтение) для каталога и выбора слота
- **Feature 3.4** — Web Admin: страница «Сотрудники»
