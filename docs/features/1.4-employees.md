## 1. Обзор

CRUD для сотрудников в Admin API. Сотрудники принадлежат бизнесу. Сотрудник связан с услугами (many-to-many): мастер может оказывать несколько услуг, услуга может оказываться несколькими мастерами. Данные сотрудников используются в Client API (Feature 1.10) для отображения в каталоге и при выборе слота записи.

### 1.1. Область реализации

| Область             | Применимо | Примечание                                      |
| ------------------- | --------- | ----------------------------------------------- |
| Backend (API)       | ☑         | NestJS, Admin API; Employee CRUD; расширение Service API (employeeIds в DTO, sync при create/update) |
| Frontend (Admin)    | ☐         | Web Admin — страница «Сотрудники» (Feature 2.4) |
| Frontend (Mini App) | ☐         | Публичный каталог (Feature 1.10)                |
| База данных         | ☑         | Prisma, Employee, EmployeeService               |

### 1.2. Контекст

| Сущность        | Назначение                                                                 |
| --------------- | -------------------------------------------------------------------------- |
| **Employee**    | Сотрудник/мастер — имя, фото, специализация. Принадлежит бизнесу             |
| **EmployeeService** | Связь many-to-many между Employee и Service (мастер ↔ услуги)          |
| **Business**    | Бизнес — владелец сотрудников                                              |
| **Service**     | Услуга — может оказываться несколькими сотрудниками                         |

## 4. Технический дизайн

### 4.1. Модель владения

Сотрудники принадлежат бизнесу (`businessId`). Доступ контролируется через `requireBusinessMember(adminUserId, businessId)`.

### 4.2. Архитектура модулей

```
apps/api/src/modules/
├── business/                    # Существующий (requireBusinessMember)
├── admin/
│   ├── admin.module.ts          # imports: AdminEmployeeModule
│   └── employee/
│       ├── admin-employee.module.ts
│       └── admin-employee.controller.ts
│
└── employee/                    # Shared (ядро)
    ├── employee.module.ts
    ├── employee.service.ts
    ├── employee.repository.ts
    └── dto/
```

**Shared-модуль EmployeeModule** — общая логика для Admin API (Feature 1.4) и Client API (Feature 1.10). Admin-слой — только контроллер с проверкой прав.

### 4.3. Связь Employee ↔ Service

- Таблица `EmployeeService` — явная связь many-to-many
- **Сотрудник:** при создании/обновлении передаётся массив `serviceIds`; полная замена связей; дубликаты дедуплицируются
- **Услуга:** при создании/обновлении (Feature 1.3) передаётся массив `employeeIds`; полная замена связей; дубликаты дедуплицируются
- Общий метод синхронизации (например, `syncEmployeeServiceLinks`) используется и при сохранении сотрудника, и при сохранении услуги
- Проверка существования и принадлежности `serviceIds`/`employeeIds` бизнесу — в одной транзакции с сохранением
- При наличии хотя бы одного невалидного ID — 404 NOT_FOUND (без раскрытия, какой именно)

### 4.4. Хранение фото

- Поле `imagePath` — путь в MinIO, например `employees/{employeeId}/photo.webp`
- StorageService: upload, delete, getPublicUrl (по аналогии с Business)
- Конвертация в WebP при загрузке

## 5. Схема базы данных

### 5.1. Таблица `Employee`

```prisma
model Employee {
  id             String   @id @default(uuid())
  businessId     String   @map("business_id")
  name           String
  specialization String?  @map("specialization")
  imagePath      String?  @map("image_path")
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  business         Business         @relation(fields: [businessId], references: [id], onDelete: Cascade)
  employeeServices EmployeeService[]

  @@index([businessId])
  @@map("employees")
}
```

### 5.2. Таблица `EmployeeService` (связь many-to-many)

```prisma
model EmployeeService {
  id         String   @id @default(uuid())
  employeeId String   @map("employee_id")
  serviceId  String   @map("service_id")
  createdAt  DateTime @default(now())

  employee Employee @relation(fields: [employeeId], references: [id], onDelete: Cascade)
  service  Service  @relation(fields: [serviceId], references: [id], onDelete: Cascade)

  @@unique([employeeId, serviceId])
  @@index([employeeId])
  @@index([serviceId])
  @@map("employee_services")
}
```

### 5.3. Обновление моделей Business и Service

В модели `Business`:
```prisma
employees Employee[]
```

В модели `Service`:
```prisma
employeeServices EmployeeService[]
```

## 6. API эндпоинты

Base path: `/api/v1/admin/businesses/:businessId`

| Метод | Путь | Описание | Доступ |
| ----- | ---- | -------- | ------ |
| GET | `/employees` | Список сотрудников (поиск, сортировка по имени) | Access token, член бизнеса |
| POST | `/employees` | Создание сотрудника | Access token, член бизнеса |
| GET | `/employees/:id` | Получение сотрудника | Access token, член бизнеса |
| PATCH | `/employees/:id` | Обновление сотрудника | Access token, член бизнеса |
| DELETE | `/employees/:id` | Удаление сотрудника | Access token, член бизнеса |
| POST | `/employees/:id/photo` | Загрузка фото | Access token, член бизнеса |
| DELETE | `/employees/:id/photo` | Удаление фото | Access token, член бизнеса |

### 6.1. Список сотрудников

**GET** `/api/v1/admin/businesses/:businessId/employees`

Без пагинации. Возвращает полный список сотрудников бизнеса. Для MVP ожидается небольшое число сотрудников на бизнес; при росте — рассмотреть пагинацию.

**Query params:**

| Параметр | Тип | Описание |
| -------- | --- | -------- |
| search | string | Поиск по имени (ILIKE). Пустая строка — игнорировать (без фильтра) |
| sort | string | Сортировка: `name` (asc), `-name` (desc). По умолчанию `name`. Невалидное значение — 400 VALIDATION_FAILED |

**Response 200:**
```json
{
  "status": "success",
  "data": [
    {
      "id": "uuid",
      "name": "Анна Иванова",
      "specialization": "Мастер маникюра",
      "imageUrl": "https://...",
      "serviceIds": ["uuid1", "uuid2"],
      "createdAt": "2024-01-15T10:00:00.000Z"
    }
  ]
}
```

При пустом списке: `{ "status": "success", "data": [] }`.

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

| Сценарий | Поведение |
| -------- | --------- |
| Бизнес не найден / нет доступа | 404 |
| Невалидный businessId (не UUID) | 400 VALIDATION_FAILED |

### 6.2. Создание сотрудника

**POST** `/api/v1/admin/businesses/:businessId/employees`

**Request body:**
```json
{
  "name": "Анна Иванова",
  "specialization": "Мастер маникюра",
  "serviceIds": ["uuid1", "uuid2"]
}
```

**Response 201:**
```json
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Анна Иванова",
    "specialization": "Мастер маникюра",
    "imageUrl": null,
    "serviceIds": ["uuid1", "uuid2"],
    "createdAt": "2024-01-15T10:00:00.000Z"
  }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND` (услуга не принадлежит бизнесу или не существует)

| Сценарий | Поведение |
| -------- | --------- |
| Один из serviceIds не принадлежит бизнесу или не найден | 404 NOT_FOUND (без раскрытия деталей) |
| serviceIds с дубликатами | дедупликация перед сохранением |
| serviceIds пустой массив | 201 — сотрудник создан без услуг |

### 6.3. Получение сотрудника

**GET** `/api/v1/admin/businesses/:businessId/employees/:id`

**Response 200:**
```json
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Анна Иванова",
    "specialization": "Мастер маникюра",
    "imageUrl": "https://...",
    "serviceIds": ["uuid1", "uuid2"],
    "createdAt": "2024-01-15T10:00:00.000Z",
    "updatedAt": "2024-01-15T10:00:00.000Z"
  }
}
```

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.4. Обновление сотрудника

**PATCH** `/api/v1/admin/businesses/:businessId/employees/:id`

Частичное обновление. Все поля опциональны. PATCH с пустым body `{}` возвращает 200 с текущими данными (без изменений).

**Request body:**
```json
{
  "name": "Анна Иванова",
  "specialization": "Мастер маникюра",
  "serviceIds": ["uuid1", "uuid2", "uuid3"]
}
```

- `serviceIds` — полная замена связей при передаче; при отсутствии поля — связи не меняются. Передача `[]` — удаление всех связей. Дубликаты дедуплицируются.
- `specialization: ""` — очистка, сохраняется как `null`.

**Response 200:** `{ "status": "success", "data": { /* обновлённый сотрудник */ } }`

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND` (услуга не принадлежит бизнесу или не существует)

### 6.5. Удаление сотрудника

**DELETE** `/api/v1/admin/businesses/:businessId/employees/:id`

**Поведение:** Hard delete. Каскадное удаление записей в `EmployeeService`. Удаление файла фото из MinIO (если есть). Если файла нет в MinIO — считать успехом (идемпотентность).

**Response 200:** `{ "status": "success", "data": { "message": "Сотрудник удалён" } }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.6. Загрузка фото сотрудника

**POST** `/api/v1/admin/businesses/:businessId/employees/:id/photo`

**Поведение:**
- `multipart/form-data`, поле `file`
- Допустимые форматы: JPEG, PNG, WebP. Проверка MIME на сервере. Недопустимый формат — 400 VALIDATION_FAILED
- Размер: max 8 MB (Multer). Превышение — 400 VALIDATION_FAILED
- Отсутствие или пустой `file` — 400 VALIDATION_FAILED
- Конвертация в WebP (sharp) → загрузка в MinIO
- При замене: удаление старого файла перед сохранением нового («последняя загрузка побеждает»)
- В БД сохраняется путь: `employees/{employeeId}/photo.webp`

**Response 200:**
```json
{
  "status": "success",
  "data": { "url": "{FILE_STORAGE_URL}/employees/{id}/photo.webp" }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.7. Удаление фото сотрудника

**DELETE** `/api/v1/admin/businesses/:businessId/employees/:id/photo`

- Удалить файл из MinIO, обновить поле в БД на `null`
- Если файла нет в MinIO — считать успехом, обновить только БД (идемпотентность)

**Response 200:** `{ "status": "success", "data": null }`

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

## 9. Безопасность и доступ

- **Guard:** AdminAuthGuard — извлечение `adminUserId` из JWT
- **Проверка:** `requireBusinessMember(adminUserId, businessId)` — для всех эндпоинтов
- **При невалидном или истёкшем JWT:** 401 `UNAUTHORIZED`
- **При отсутствии доступа:** 404 `NOT_FOUND`
- **При отсутствии доступа к ресурсу (employee):** 404 `NOT_FOUND` (не раскрывать существование)

**Env:** `FILE_STORAGE_URL`, `FILE_STORAGE_BUCKET` — по аналогии с Feature 1.2.

## 11. Конфигурация и зависимости

**Env-переменные:**

- `FILE_STORAGE_URL` — базовый URL хранилища без завершающего слэша
- `FILE_STORAGE_BUCKET` — имя bucket (по умолчанию `uploads`)

**Зависимости:**

- **Feature 1.1** — аутентификация AdminUser (AdminAuthGuard)
- **Feature 1.2** — Business, BusinessService.requireBusinessMember
- **Feature 1.3** — Service, ServiceModule. **В объём 1.4 входит:** (1) добавление `employeeIds` в `CreateServiceDto` и `UpdateServiceDto`; (2) вызов метода синхронизации из `ServiceService` при create/update; (3) возврат `employeeIds` в ответах списка и получения услуги
- **StorageModule** — StorageService (upload, delete, WebP)
- **Prisma** — Employee, EmployeeService
