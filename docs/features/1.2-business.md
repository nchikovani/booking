# Feature 1.2: Бизнес — CRUD и настройки компании (Admin API)

**Описание реализации**  
**Связанные документы:** [Architecture](../architecture.md), [Roadmap](../roadmap.md), [PRD](../prd.md)

---

## 1. Обзор

Управление бизнесом (компанией) в Admin API. Владелец или администратор создаёт и редактирует информацию о своей компании через Web Admin. Данные компании отображаются клиентам в Telegram Mini App (Feature 1.9).

### 1.1. Контекст


| Сущность      | Назначение                                                                  |
| ------------- | --------------------------------------------------------------------------- |
| **Business**  | Бизнес/организация — салон, студия, мастер и т.п. Хранит настройки компании |
| **AdminUser** | Владелец или администратор, привязанный к одному или нескольким бизнесам    |


Данный документ описывает CRUD операций с Business и настройки компании в контексте Admin API.

---

## 2. Связь AdminUser и Business

### 2.1. Модель владения

На этапе MVP: **один AdminUser — один Business**. Бизнес создаётся **автоматически при первом входе** (при вызове `GET /me` — см. [Feature 1.1](1.1-auth.md)). Отдельного эндпоинта для создания бизнеса нет.

Модель `BusinessMember` сохраняется для будущего расширения (несколько бизнесов на владельца). Ограничение «максимум 1 бизнес» реализуется на этапе MVP.

Связь: `AdminUser` ↔ `Business` через таблицу `BusinessMember`. **На этапе MVP — только роль OWNER.**

---

## 3. Архитектура модуля

Данные о бизнесе используются в **двух контекстах**:
- **Admin API** — управление бизнесом (CRUD, загрузка файлов)
- **Client API** (Mini App) — публичная информация о компании (Feature 1.9)

Для переиспользования логики создаётся **общий (shared) BusinessModule**, который импортируется и Admin, и App модулями.

│   └── ...
│
├── admin/

│   ├── business/               # Admin-специфичный слой
│   │   ├── admin-business.module.ts
│   │   ├── admin-business.controller.ts   # /api/v1/admin/businesses
│   │   └── admin-business-file.controller.ts  # или в том же controller
│   └── auth/
│       └── ...
│
└── app/                        # Client API (Feature 1.9) — структура только в документации

*В Feature 1.2 создаётся только `business/` и `admin/business/`. `modules/app/` не создаётся — описание для Feature 1.9.*

### 3.2. Разделение ответственности

| Компонент | Ответственность |
| --------- | ---------------- |
| **BusinessService** | CRUD, ensureBusinessForUser, findById (с проверкой доступа или без) |
| **AdminBusinessController** | Эндпоинты Admin API, проверка членства (BusinessMember), OWNER для DELETE |
| **AppBusinessController** | Публичный эндпоинт для Mini App — только чтение, без авторизации Admin |

BusinessService предоставляет методы:
- `ensureBusinessForUser(adminUserId)` — создание бизнеса при первом входе
- `findById(id)` — для внутреннего использования
- `findByIdPublic(id)` — публичные данные (без createdAt и т.п.) для Mini App
- `findByAdminUser(adminUserId)` — список бизнесов пользователя
- `update(id, dto)` — обновление
- `delete(id)` — удаление (с проверкой OWNER в контроллере)

## 4. Схема базы данных

### 4.1. Таблица `Business`

Основная сущность компании.

```prisma
model Business {
  id          String   @id @default(uuid())
  name        String?  // название компании
  description String?  @db.Text  // описание
  logoPath    String?  @map("logo_path")  // путь: businesses/{id}/logo.webp
  imagePath   String?  @map("image_path")  // путь: businesses/{id}/image.webp

  // Контакты
  phone       String?
  email       String?
  website     String?
  telegram    String?  // username или ссылка
  vk          String?
  youtube     String?

  // Адрес
  address     String?  @db.Text
  latitude    Float?
  longitude   Float?

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  businessMembers  BusinessMember[]

  @@map("businesses")
}
```

### 4.2. Таблица `BusinessMember`

Связь AdminUser с Business. **На этапе MVP — только OWNER.** ADMIN зарезервирован для будущего расширения.

```prisma
enum BusinessRole {
  OWNER   // владелец — полный доступ (MVP; ADMIN — в будущем)
}

model BusinessMember {
  id          String       @id @default(uuid())
  adminUserId String       @map("admin_user_id")
  businessId  String       @map("business_id")
  role        BusinessRole @default(OWNER)
  createdAt   DateTime     @default(now())

  adminUser   AdminUser    @relation(fields: [adminUserId], references: [id], onDelete: Cascade)
  business    Business     @relation(fields: [businessId], references: [id], onDelete: Cascade)

  @@unique([adminUserId, businessId])
  @@index([businessId])
  @@map("business_members")
}
```

В модели `AdminUser` добавляется: `businessMembers BusinessMember[]`

### 4.3. Режим работы (опционально)

Режим работы компании (общий для бизнеса) может храниться в отдельной таблице `BusinessSchedule` или в JSON-поле. Для MVP допустимо хранить в `Business` как JSON или отложить до Feature 1.5 (расписание сотрудников). Детали — в Feature 1.5.

## 5. API эндпоинты (Admin)

Base path: `/api/v1/admin/businesses`

Все эндпоинты требуют **AdminAuthGuard** (access token). Доступ только к бизнесам, в которых пользователь является членом (BusinessMember).


| Метод  | Путь   | Описание                              | Доступ                     |
| ------ | ------ | ------------------------------------- | -------------------------- |
| GET    | `/`    | Список бизнесов текущего пользователя | Access token               |
| GET    | `/:id` | Получение бизнеса по ID               | Access token, член бизнеса |
| PATCH  | `/:id` | Обновление бизнеса                    | Access token, член бизнеса |
| DELETE | `/:id` | Удаление бизнеса                      | Access token, OWNER        |
| POST   | `/:id/logo`  | Загрузка логотипа                 | Access token, член бизнеса |
| POST   | `/:id/image` | Загрузка изображения/обложки       | Access token, член бизнеса |
| DELETE | `/:id/logo`  | Удаление логотипа                 | Access token, член бизнеса |
| DELETE | `/:id/image` | Удаление изображения/обложки       | Access token, член бизнеса |


### 5.1. Список бизнесов

**GET** `/api/v1/admin/businesses`

Возвращает бизнесы, в которых текущий AdminUser является членом. На этапе MVP — 0 или 1 элемент. Пустой массив, если бизнес ещё не создан (будет создан при следующем вызове `GET /me`).

```json
// Response 200
{
  "status": "success",
  "data": [
    {
      "id": "uuid",
      "name": "Салон красоты «Элегант»",
      "description": "Салон красоты в центре города",
      "logoUrl": "https://...",  // полный URL (FILE_STORAGE_URL + logoPath)
      "role": "OWNER"
    }
  ]
}
```

### 5.2. Получение бизнеса

**GET** `/api/v1/admin/businesses/:id`

Возвращает полные данные бизнеса. Доступ только для членов бизнеса.

```json
// Response 200
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Салон красоты «Элегант»",
    "description": "...",
    "logoUrl": "https://...",  // полный URL (FILE_STORAGE_URL + logoPath)
    "imageUrl": null,           // полный URL (FILE_STORAGE_URL + imagePath)
    "phone": "+7 999 123-45-67",
    "email": "info@elegant.ru",
    "website": "https://elegant.ru",
    "telegram": "@elegant_salon",
    "vk": null,
    "youtube": "https://youtube.com/@elegant_salon",
    "address": "г. Москва, ул. Примерная, д. 1",
    "latitude": 55.75,
    "longitude": 37.62,
    "createdAt": "...",
    "updatedAt": "..."
  }
}
```

### 5.3. Обновление бизнеса

**PATCH** `/api/v1/admin/businesses/:id`

Частичное обновление. Все поля опциональны. **Пустая строка `""` для опционального поля** трактуется как очистка и сохраняется как `null`.

```json
// Request
{
  "name": "Салон красоты «Элегант» (обновлено)",
  "description": "Новое описание",
  "website": "https://elegant.ru",
  "youtube": "https://youtube.com/@elegant_salon"
}

// Response 200
{
  "status": "success",
  "data": { /* обновлённый бизнес */ }
}
```

### 5.4. Удаление бизнеса

**DELETE** `/api/v1/admin/businesses/:id`

Только для роли OWNER. **Hard delete** с каскадным удалением связанных сущностей (услуги, сотрудники, записи, клиенты).

```json
// Response 200
{
  "status": "success",
  "data": { "message": "Бизнес удалён" }
}
```

## 6. Загрузка файлов (логотип, изображение)

### 6.1. StorageModule и StorageService

**StorageModule** — общий модуль для работы с файлами. Содержит **StorageService**:

- **Метод `upload(buffer, key, options?)`** — загрузка в MinIO, возврат **пути** (key) для сохранения в БД
- **Метод `delete(key)`** — удаление файла из MinIO
- **Структура ключей:** `businesses/{businessId}/logo.webp`, `businesses/{businessId}/image.webp` (сохраняются в `logoPath`, `imagePath`)
- **Валидация:** размер (max 5 MB), форматы на входе (JPEG, PNG, WebP)
- **Конвертация в WebP** — все загружаемые изображения конвертируются в WebP для экономии трафика и единообразия

### 6.2. Прямые URL к хранилищу

В БД хранится **путь** (key) в полях `logoPath`, `imagePath`: `businesses/{id}/logo.webp`. Публичный URL формируется как `{FILE_STORAGE_URL}/{path}`.

- **Env:** `FILE_STORAGE_URL` — базовый URL хранилища без завершающего слэша (dev: `http://localhost:9000/uploads`, prod: MinIO или CDN)
- **Env:** `FILE_STORAGE_BUCKET` — имя bucket (по умолчанию `uploads`)
- **При подключении CDN:** меняется только `FILE_STORAGE_URL` на URL CDN, код не меняется
- **В ответах API** (`logoUrl`, `imageUrl`): полный URL = `{FILE_STORAGE_URL}/{path}` (формируется при сериализации)

### 6.3. Создание bucket при старте API

Bucket создаётся **при старте приложения** (StorageService или StorageModule). При инициализации:

- Проверить существование bucket (MinIO Client API: `bucketExists`)
- Если bucket не существует — создать (`makeBucket`)
- Установить bucket policy для публичного чтения (`GetObject` для `*`)
- При ошибке — логировать, не блокировать старт (MinIO может быть недоступен в dev)

### 6.4. Публичный доступ к MinIO

- Bucket с политикой `GetObject` для `*` (анонимное чтение)
- Файлы доступны по URL `{FILE_STORAGE_URL}/{path}` без аутентификации
- CORS: настроить `Access-Control-Allow-Origin` для доменов Web Admin и Mini App

### 6.5. Конвертация в WebP

- **Библиотека:** `sharp` — быстрая, поддерживает JPEG/PNG → WebP
- **Параметры:** quality 85, без изменения размера (или max width 1920 для image)
- **Вход:** JPEG, PNG, WebP (WebP пропускается без конвертации)
- **Выход:** всегда `.webp`

### 6.6. Эндпоинты загрузки

**POST** `/api/v1/admin/businesses/:id/logo`  
**POST** `/api/v1/admin/businesses/:id/image`

- `multipart/form-data`, поле `file`
- Конвертация в WebP (sharp) → загрузка в MinIO через StorageService
- **При замене:** удаление старого файла из MinIO (StorageService.delete) перед сохранением нового
- В БД сохраняется путь в `logoPath` / `imagePath`: `businesses/{id}/logo.webp`, `businesses/{id}/image.webp`
- Response 200: `{ "status": "success", "data": { "url": "{FILE_STORAGE_URL}/businesses/{id}/logo.webp" } }`

### 6.7. Эндпоинты удаления логотипа/изображения

**DELETE** `/api/v1/admin/businesses/:id/logo`  
**DELETE** `/api/v1/admin/businesses/:id/image`

- Удалить файл из MinIO (StorageService.delete) по текущему пути
- Обновить поле `logoPath` / `imagePath` в БД на `null`
- Если файла нет в MinIO — считать успехом, обновить только БД (идемпотентность)

| address     | string | max 500        |

## 8. Проверка прав доступа

- **Guard:** AdminAuthGuard — извлечение `adminUserId` из JWT
- **Проверка членства:** для GET/PATCH/DELETE `/:id` и для POST/DELETE logo/image — запрос в BusinessMember по `adminUserId` и `businessId`
- **DELETE:** только OWNER (в MVP — единственная роль)
- **При отсутствии доступа к бизнесу:** 404 `NOT_FOUND` (не раскрывать существование бизнеса)

## 9. Edge cases


| Сценарий                  | Поведение                                      |
| ------------------------- | ---------------------------------------------- |
| Бизнес не найден          | 404 `NOT_FOUND`                                |
| Нет доступа к бизнесу     | 404 `NOT_FOUND` (не раскрывать существование)  |
| DELETE не OWNER           | 404 `NOT_FOUND`                                |
| Некорректное имя (PATCH)  | 400 `VALIDATION_FAILED` (если > 200 символов)  |
| Некорректный файл (загрузка) | 400 `VALIDATION_FAILED` (формат, размер)    |
| DELETE logo/image при отсутствии файла | 200 OK (обновить БД, идемпотентно)      |
