# Feature 1.2: Бизнес — CRUD и настройки компании (Admin API)

**Описание реализации**  
**Связанные документы:** [Architecture](../architecture.md), [Roadmap](../roadmap.md), [Feature 1.1](1.1-auth.md)

## 1. Обзор

Управление бизнесом (компанией) в Admin API. Владелец или администратор создаёт и редактирует информацию о своей компании через Web Admin. Данные компании отображаются клиентам в Telegram Mini App (Feature 2.2).

### 1.1. Область реализации

| Область             | Применимо | Примечание                                      |
| ------------------- | --------- | ----------------------------------------------- |
| Backend (API)       | ☑         | NestJS, BusinessModule, Admin API эндпоинты     |
| Frontend (Admin)    | ☐         | Web Admin — страница «Настройки компании» (Feature 3.2) |
| Frontend (Mini App) | ☐         | Публичный просмотр (Feature 2.2)               |
| База данных         | ☑         | Prisma, Business, BusinessMember                |

### 1.2. Контекст

| Сущность      | Назначение                                                                  |
| ------------- | --------------------------------------------------------------------------- |
| **Business**  | Бизнес/организация — салон, студия, мастер и т.п. Хранит настройки компании |
| **AdminUser** | Владелец или администратор, привязанный к одному или нескольким бизнесам    |

На этапе MVP: **один AdminUser — один Business**. Бизнес создаётся **автоматически при первом входе** (при вызове `GET /me` — см. [Feature 1.1](1.1-auth.md)). Отдельного эндпоинта для создания бизнеса нет.

## 3. Критерии приёмки

- [ ] GET /businesses возвращает список бизнесов текущего пользователя
- [ ] GET /businesses/:id возвращает полные данные (только для членов)
- [ ] PATCH /businesses/:id обновляет бизнес (частичное обновление)
- [ ] DELETE /businesses/:id удаляет бизнес (только OWNER)
- [ ] POST /businesses/:id/logo и /image загружают файлы в MinIO (WebP)
- [ ] DELETE /businesses/:id/logo и /image удаляют файлы
- [ ] requireBusinessMember / requireBusinessOwner проверяют доступ
- [ ] При отсутствии доступа — 404 NOT_FOUND

## 4. Технический дизайн

### 4.1. Модель владения

Связь: `AdminUser` ↔ `Business` через таблицу `BusinessMember`. **На этапе MVP — только роль OWNER.** Модель `BusinessMember` сохранена для будущего расширения (несколько бизнесов на владельца).

### 4.2. Архитектура модуля

Данные о бизнесе используются в **двух контекстах**:
- **Admin API** — управление бизнесом (CRUD, загрузка файлов)
- **Client API** (Mini App) — публичная информация о компании (Feature 2.2)

Создаётся **общий (shared) BusinessModule**, который импортируется и Admin, и App модулями.

### 4.3. Структура модулей


### 4.4. Разделение ответственности

| Компонент | Ответственность |
| --------- | ---------------- |
| **BusinessService** | CRUD, ensureBusinessForUser, findById, findByIdPublic, findByAdminUser, update, delete, uploadLogo/Image, deleteLogo/Image |
| **AdminBusinessController** | Эндпоинты Admin API, проверка через `requireBusinessMember`/`requireBusinessOwner` |
| **AppBusinessController** | Публичный эндпоинт для Mini App — только чтение, без авторизации Admin |

BusinessService предоставляет методы:
- `ensureBusinessForUser(adminUserId)` — создание бизнеса при первом входе
- `requireBusinessMember(adminUserId, businessId)` — проверка членства, 404 при отсутствии
- `requireBusinessOwner(adminUserId, businessId)` — проверка роли OWNER, 404 при отсутствии
- `findById(id)`, `findByIdPublic(id)`, `findByAdminUser(adminUserId)`, `update(id, dto)`, `delete(id)`
- `uploadLogo(businessId, buffer)`, `uploadImage(businessId, buffer)`, `deleteLogo(businessId)`, `deleteImage(businessId)`

### 4.5. StorageModule и StorageService

- **upload(buffer, key, options?)** — загрузка в MinIO, возврат **пути** (key) для сохранения в БД
- **delete(key)** — удаление файла из MinIO
- **Структура ключей:** `businesses/{businessId}/logo.webp`, `businesses/{businessId}/image.webp`
- **Валидация:** размер (max 8 MB), форматы (JPEG, PNG, WebP)
- **Конвертация в WebP** — sharp, quality 85
- **Bucket:** создаётся при старте (StorageInitService, OnModuleInit)
- **Публичный доступ:** bucket policy `GetObject` для `*`

## 5. Схема базы данных

### 5.1. Таблица `Business`

```prisma
model Business {
  id          String   @id @default(uuid())
  name        String?  // название компании
  description String?  @db.Text  // описание
  logoPath    String?  @map("logo_path")  // путь: businesses/{id}/logo.webp
  imagePath   String?  @map("image_path")  // путь: businesses/{id}/image.webp

  // Контакты
  phone       String?
  email       String?
  website     String?
  telegram    String?  // username или ссылка
  vk          String?
  youtube     String?

  // Адрес
  address     String?  @db.Text
  latitude    Float?
  longitude   Float?

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  businessMembers  BusinessMember[]

  @@map("businesses")
}
```

### 5.2. Таблица `BusinessMember` и enum `BusinessRole`

```prisma
enum BusinessRole {
  OWNER   // владелец — полный доступ (MVP; ADMIN — в будущем)
}

model BusinessMember {
  id          String       @id @default(uuid())
  adminUserId String       @map("admin_user_id")
  businessId  String       @map("business_id")
  role        BusinessRole @default(OWNER)
  createdAt   DateTime     @default(now())

  adminUser   AdminUser    @relation(fields: [adminUserId], references: [id], onDelete: Cascade)
  business    Business     @relation(fields: [businessId], references: [id], onDelete: Cascade)

  @@unique([adminUserId, businessId])
  @@index([businessId])
  @@map("business_members")
}
```

В модели `AdminUser` добавляется: `businessMembers BusinessMember[]`

## 6. API эндпоинты

Base path: `/api/v1/admin/businesses`

| Метод  | Путь   | Описание                              | Доступ                     |
| ------ | ------ | ------------------------------------- | -------------------------- |
| GET    | `/`    | Список бизнесов текущего пользователя | Access token               |
| GET    | `/:id` | Получение бизнеса по ID               | Access token, член бизнеса |
| PATCH  | `/:id` | Обновление бизнеса                    | Access token, член бизнеса |
| DELETE | `/:id` | Удаление бизнеса                      | Access token, OWNER        |
| POST   | `/:id/logo`  | Загрузка логотипа                 | Access token, член бизнеса |
| POST   | `/:id/image` | Загрузка изображения/обложки       | Access token, член бизнеса |
| DELETE | `/:id/logo`  | Удаление логотипа                 | Access token, член бизнеса |
| DELETE | `/:id/image` | Удаление изображения/обложки       | Access token, член бизнеса |

### 6.1. Список бизнесов

**GET** `/api/v1/admin/businesses`

Возвращает бизнесы, в которых текущий AdminUser является членом. На этапе MVP — 0 или 1 элемент. Пустой массив, если бизнес ещё не создан (будет создан при следующем вызове `GET /me`).

**Response 200:**
```json
{
  "status": "success",
  "data": [
    {
      "id": "uuid",
      "name": "Салон красоты «Элегант»",
      "description": "Салон красоты в центре города",
      "logoUrl": "https://...",
      "role": "OWNER"
    }
  ]
}
```

**Errors:** `401 UNAUTHORIZED`

### 6.2. Получение бизнеса

**GET** `/api/v1/admin/businesses/:id`

**Response 200:**
```json
{
  "status": "success",
  "data": {
    "id": "uuid",
    "name": "Салон красоты «Элегант»",
    "description": "...",
    "logoUrl": "https://...",
    "imageUrl": null,
    "phone": "+7 999 123-45-67",
    "email": "info@elegant.ru",
    "website": "https://elegant.ru",
    "telegram": "@elegant_salon",
    "vk": null,
    "youtube": "https://youtube.com/@elegant_salon",
    "address": "г. Москва, ул. Примерная, д. 1",
    "latitude": 55.75,
    "longitude": 37.62,
    "createdAt": "...",
    "updatedAt": "..."
  }
}
```

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND`

| Сценарий | Поведение |
| -------- | --------- |
| Бизнес не найден | 404 `NOT_FOUND` |
| Нет доступа к бизнесу | 404 `NOT_FOUND` (не раскрывать существование) |

### 6.3. Обновление бизнеса

**PATCH** `/api/v1/admin/businesses/:id`

Частичное обновление. Все поля опциональны. **Пустая строка `""`** для опционального поля трактуется как очистка и сохраняется как `null`.

**Request body:**
```json
{
  "name": "Салон красоты «Элегант» (обновлено)",
  "description": "Новое описание",
  "website": "https://elegant.ru",
  "youtube": "https://youtube.com/@elegant_salon"
}
```

**Response 200:** `{ "status": "success", "data": { /* обновлённый бизнес */ } }`

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.4. Удаление бизнеса

**DELETE** `/api/v1/admin/businesses/:id`

Только для роли OWNER. **Hard delete** с каскадным удалением связанных сущностей.

**Response 200:**
```json
{
  "status": "success",
  "data": { "message": "Бизнес удалён" }
}
```

**Errors:** `401 UNAUTHORIZED` | `404 NOT_FOUND` (в т.ч. если не OWNER)

### 6.5. Загрузка логотипа и изображения

**POST** `/api/v1/admin/businesses/:id/logo`  
**POST** `/api/v1/admin/businesses/:id/image`

- `multipart/form-data`, поле `file`
- Multer: лимит 8 MB
- Конвертация в WebP (sharp) → загрузка в MinIO
- При замене: удаление старого файла перед сохранением нового
- В БД сохраняется путь: `businesses/{id}/logo.webp`, `businesses/{id}/image.webp`

**Response 200:**
```json
{
  "status": "success",
  "data": { "url": "{FILE_STORAGE_URL}/businesses/{id}/logo.webp" }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 UNAUTHORIZED` | `404 NOT_FOUND`

### 6.6. Удаление логотипа и изображения

**DELETE** `/api/v1/admin/businesses/:id/logo`  
**DELETE** `/api/v1/admin/businesses/:id/image`

- Удалить файл из MinIO, обновить поле в БД на `null`
- Если файла нет в MinIO — считать успехом, обновить только БД (идемпотентность)

**Response 200:** `{ "status": "success" }`

## 9. Безопасность и доступ

- **Guard:** AdminAuthGuard — извлечение `adminUserId` из JWT
- **Проверка членства:** `requireBusinessMember(adminUserId, businessId)` — для GET/PATCH, POST/DELETE logo/image
- **DELETE бизнеса:** `requireBusinessOwner(adminUserId, businessId)` — только OWNER
- **При отсутствии доступа:** 404 `NOT_FOUND` (не раскрывать существование бизнеса)

**Env:** `FILE_STORAGE_URL` — базовый URL хранилища (dev: `http://localhost:9000/uploads`, prod: MinIO или CDN). `FILE_STORAGE_BUCKET` — имя bucket (по умолчанию `uploads`).

## 11. Конфигурация и зависимости

**Env-переменные:**

- `FILE_STORAGE_URL` — базовый URL хранилища без завершающего слэша
- `FILE_STORAGE_BUCKET` — имя bucket (по умолчанию `uploads`)

**Зависимости:**

- **Feature 1.1** — аутентификация AdminUser (AdminAuthGuard)
- **StorageModule** — StorageService (upload, delete, WebP), StorageInitService (bucket при старте)
- **Prisma** — Business, BusinessMember, enum BusinessRole

**Интеграция с Feature 1.1:** `GET /me` вызывает `BusinessService.ensureBusinessForUser(adminUserId)` — при отсутствии бизнеса создаётся Business и BusinessMember (role: OWNER). AdminAuthModule импортирует BusinessModule.
