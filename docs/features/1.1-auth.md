# Feature 1.1: Аутентификация и авторизация (Admin API)

**Этап:** Admin API 1.1  
**Связанные документы:** [Architecture](../architecture.md), [Roadmap](../roadmap.md)

## 1. Обзор

Реализация аутентификации и авторизации для Web Admin. Пользователи (владельцы бизнесов, администраторы) входят в систему через учётные данные и получают доступ к Admin API на основе токенов.

### 1.1. Разделение типов пользователей

В системе предусмотрены **два типа пользователей**, каждый со своей таблицей и своим потоком аутентификации:


| Тип            | Таблица     | Контекст                                              | Аутентификация                        |
| -------------- | ----------- | ----------------------------------------------------- | ------------------------------------- |
| **Admin User** | `AdminUser` | Web Admin — владельцы бизнесов, администраторы        | Email + пароль, токены (текущая фича) |
| **Client**     | `Client`    | Telegram Mini App — клиенты, записывающиеся на услуги | Telegram initData (Feature 1.8)       |


Текущая фича 1.1 касается **только Admin User**. Схема ниже описывает таблицы для админки. Таблица `Client` и её аутентификация будут реализованы отдельно (см. roadmap, этап 1.8).

На текущую реализацию это не влияет: модуль auth, эндпоинты и guards изолированы в контексте Admin API. В будущем Client API получит свой auth-модуль с проверкой Telegram initData.

## 2. Архитектура аутентификации

### 2.1. Расширяемость способов входа

Текущая реализация: **только email + пароль**.

Архитектура предусматривает добавление других способов в будущем (OAuth, magic link и т.д.):

```
┌─────────────────────────────────────────────────────────────┐
│                    AuthStrategy (интерфейс)                  │
├─────────────────────────────────────────────────────────────┤
│  + authenticate(credentials): Promise<AuthResult>           │
└─────────────────────────────────────────────────────────────┤
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ EmailPassword   │  │ OAuthGoogle      │  │ MagicLink       │
│ Strategy        │  │ Strategy (future)│  │ (future)        │
└─────────────────┘  └─────────────────┘  └─────────────────┘
```

- **Strategy pattern** — каждый способ входа реализует общий интерфейс
- **AuthProvider** в БД — привязка учётной записи к способу входа
- Регистрация стратегий через DI (NestJS)

### 2.2. Токены


| Токен            | Назначение                     | Время жизни | Передача                                  |
| ---------------- | ------------------------------ | ----------- | ----------------------------------------- |
| **accessToken**  | Доступ к защищённым эндпоинтам | 15 мин      | Заголовок `Authorization: Bearer <token>` |
| **refreshToken** | Обновление accessToken         | 7 дней      | HttpOnly cookie (защита от XSS)           |


Клиент хранит accessToken в памяти и отправляет в каждом запросе в `Authorization`. Refresh token — только в cookie, клиент не имеет к нему доступа. Для `/refresh` запросы должны отправляться с `credentials: 'include'`.

**Флоу при загрузке приложения:** accessToken теряется при перезагрузке. При загрузке: если есть refresh cookie (клиент не может проверить HttpOnly, но может попытаться вызвать `/refresh`), вызвать `/refresh` и получить новый accessToken. Если 401 — редирект на логин.

**Формат:** JWT для обоих токенов (или opaque для refresh — см. раздел 5).

**Refresh token rotation:** при каждом обновлении выдаётся новый refreshToken, старый инвалидируется.

**Количество сессий:** не ограничивается — каждый вход создаёт новую сессию (refresh token).

## 3. Схема базы данных

Все таблицы ниже относятся **только к Admin User**. Таблица `Client` для Mini App — отдельная сущность (см. Feature 1.8). Существующая модель `User` в Prisma удаляется, данных для миграции нет.

### 3.1. Таблица `AdminUser`

Профиль пользователя админки (владельцы бизнесов, администраторы).

```prisma
model AdminUser {
  id        String   @id @default(uuid())
  email     String?  @unique  // опционально: OAuth может не вернуть email; для EMAIL_PASSWORD — обязателен (валидация в DTO)
  firstName String?
  lastName  String?
  emailVerifiedAt DateTime?   // для будущей верификации email
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  authProviders       AuthProvider[]
  refreshTokens        RefreshToken[]
  passwordResetTokens  PasswordResetToken[]
}
```

### 3.2. Таблица `AuthProvider`

Связь пользователя со способом входа. Один пользователь может иметь несколько провайдеров (email + Google в будущем).

```prisma
enum AuthProviderType {
  EMAIL_PASSWORD
  // GOOGLE, MAGIC_LINK — в будущем
}

model AuthProvider {
  id            String          @id @default(uuid())
  adminUserId   String          @map("admin_user_id")
  type          AuthProviderType
  providerId    String?         @map("provider_id")  // для OAuth: external id
  createdAt     DateTime        @default(now())

  adminUser     AdminUser       @relation(fields: [adminUserId], references: [id], onDelete: Cascade)
  credential    UserCredential?  // только для EMAIL_PASSWORD

  @@unique([adminUserId, type])
  @@map("auth_providers")
}
```

### 3.3. Таблица `UserCredential`

Хранение хеша пароля (только для `EMAIL_PASSWORD`).

```prisma
model UserCredential {
  id             String   @id @default(uuid())
  authProviderId String   @unique @map("auth_provider_id")
  passwordHash   String   @map("password_hash")  // bcrypt/argon2
  updatedAt      DateTime @updatedAt

  authProvider   AuthProvider @relation(fields: [authProviderId], references: [id], onDelete: Cascade)

  @@map("user_credentials")
}
```

### 3.4. Таблица `RefreshToken`

Хранение refresh-токенов для валидации и отзыва.

```prisma
model RefreshToken {
  id           String    @id @default(uuid())
  adminUserId  String    @map("admin_user_id")
  tokenHash    String    @unique @map("token_hash")  // SHA-256 от токена
  deviceInfo   String?   @map("device_info")        // User-Agent, для аудита
  expiresAt    DateTime  @map("expires_at")
  revokedAt    DateTime? @map("revoked_at")
  replacedBy   String?   @map("replaced_by")        // id нового токена при rotation
  createdAt    DateTime  @default(now())

  adminUser    AdminUser @relation(fields: [adminUserId], references: [id], onDelete: Cascade)

  @@index([adminUserId])
  @@index([tokenHash])
  @@index([expiresAt])  // для очистки истёкших
  @@map("refresh_tokens")
}
```

### 3.5. Таблица `LoginAttempt`

Ограничение попыток входа и блокировка при brute-force.

```prisma
model LoginAttempt {
  id        String   @id @default(uuid())
  email     String   // нормализованный
  ipAddress String?  @map("ip_address")
  success   Boolean
  createdAt DateTime @default(now())

  @@index([email, createdAt])
  @@index([ipAddress, createdAt])
  @@map("login_attempts")
}
```

### 3.6. Таблица `PasswordResetToken` (для forgot/reset password)

```prisma
model PasswordResetToken {
  id        String   @id @default(uuid())
  adminUserId String @map("admin_user_id")
  tokenHash  String   @unique @map("token_hash")  // SHA-256 от токена
  expiresAt  DateTime @map("expires_at")
  usedAt     DateTime? @map("used_at")
  createdAt  DateTime @default(now())

  adminUser  AdminUser @relation(fields: [adminUserId], references: [id], onDelete: Cascade)

  @@index([adminUserId])
  @@index([tokenHash])
  @@index([expiresAt])
  @@map("password_reset_tokens")
}
```

В модели `AdminUser` добавить: `passwordResetTokens PasswordResetToken[]`

## 4. API эндпоинты

Base path: `/api/v1/admin/auth`


| Метод | Путь               | Описание                           | Доступ                       |
| ----- | ------------------ | ---------------------------------- | ---------------------------- |
| POST  | `/register`        | Регистрация (email + пароль)       | Public                       |
| POST  | `/login`           | Вход (email + пароль)              | Public                       |
| POST  | `/refresh`         | Обновление accessToken             | Refresh cookie               |
| POST  | `/logout`          | Выход (инвалидация текущей сессии) | Refresh cookie (обязательно) |
| GET   | `/me`              | Текущий пользователь               | Access token (Authorization) |
| POST  | `/forgot-password` | Запрос сброса пароля               | Public                       |
| POST  | `/reset-password`  | Сброс пароля по токену             | Public                       |


**Регистрация:** открытая — любой может зарегистрироваться.

### 4.1. Регистрация

**POST** `/api/v1/admin/auth/register`

Создание AdminUser, AuthProvider, UserCredential, RefreshToken — **в одной транзакции** (Prisma `$transaction`). Иначе при сбое возможны частично созданные данные.

```json
// Request
{
  "email": "user@example.com",
  "password": "SecureP@ssw0rd",
  "firstName": "Иван",
  "lastName": "Иванов"
}

// Response 201
// refreshToken в Set-Cookie (HttpOnly), accessToken и user в body
{
  "status": "success",
  "data": {
    "user": {
      "id": "uuid",
      "email": "user@example.com",
      "firstName": "Иван",
      "lastName": "Иванов"
    },
    "accessToken": "eyJ...",
    "expiresIn": 900
  }
}
```

### 4.2. Вход

**POST** `/api/v1/admin/auth/login`

```json
// Request
{
  "email": "user@example.com",
  "password": "SecureP@ssw0rd"
}

// Response 200
// refreshToken в Set-Cookie (HttpOnly), accessToken и user в body
{
  "status": "success",
  "data": {
    "user": { "id", "email", "firstName", "lastName" },
    "accessToken": "eyJ...",
    "expiresIn": 900
  }
}
```

### 4.3. Обновление токена

**POST** `/api/v1/admin/auth/refresh`

- Cookie: `refreshToken` (все пути с refresh token — под `/api/v1/admin/auth`)
- Response: тот же формат, что и при login (refreshToken в Set-Cookie, accessToken в body)

### 4.4. Выход

**POST** `/api/v1/admin/auth/logout`

- Cookie: `refreshToken` (обязательно) — инвалидирует текущую сессию
- Response: `{ "status": "success" }` + `Set-Cookie: refreshToken=; Max-Age=0; Path=/api/v1/admin/auth` — очистка cookie (Path должен совпадать с Path при установке). При заданном COOKIE_DOMAIN добавлять Domain=COOKIE_DOMAIN
- Если cookie отсутствует: 401 `UNAUTHORIZED` (сессия не идентифицирована)

### 4.5. Забытый пароль

**POST** `/api/v1/admin/auth/forgot-password`

```json
// Request
{
  "email": "user@example.com"
}

// Response 200 (всегда — не раскрывать наличие email)
{
  "status": "success",
  "data": { "message": "Если аккаунт существует, на email отправлена ссылка для сброса пароля" }
}
```

- Токен создаётся **только если** найден AdminUser с AuthProvider типа `EMAIL_PASSWORD` (для OAuth-only пользователей сброс пароля недоступен)
- Генерируется одноразовый токен (random в БД, SHA-256 в `tokenHash`)
- Ссылка вида `{FRONTEND_URL}/reset-password?token=...` — `FRONTEND_URL` из env
- Токен действителен 1 час
- При новом запросе для того же email: **удалить** все `PasswordResetToken` этого пользователя с `usedAt = null` (активна только последняя ссылка)

### 4.6. Сброс пароля

**POST** `/api/v1/admin/auth/reset-password`

```json
// Request
{
  "token": "одноразовый токен из ссылки",
  "password": "NewSecureP@ssw0rd"
}

// Response 200
{
  "status": "success",
  "data": { "message": "Пароль успешно изменён" }
}
```

- После успешного сброса: токен инвалидируется (`usedAt`), **все refresh token пользователя** отзываются (выход со всех устройств)
- Edge cases: неверный/истёкший/уже использованный токен → 400, `INVALID_OR_EXPIRED_TOKEN`

## 5. Безопасность

### 5.1. Пароли

- **Хеширование:** Argon2id (или bcrypt) с солью
- **Минимальные требования:** 8+ символов, буквы + цифры, опционально спецсимволы
- **Валидация:** класс-validator в DTO

### 5.2. Токены

- **AccessToken (JWT):** подпись HS256/RS256, payload: `{ sub: adminUserId, iat, exp, type: "access" }`
- **RefreshToken:** 
  - Вариант A: JWT с `type: "refresh"`, проверка в БД по `tokenHash`
  - Вариант B: opaque token (random 64 bytes), только в БД
- **Refresh rotation:** при каждом `/refresh` — новый refresh, старый помечается `revokedAt`, `replacedBy`

### 5.3. Защита от атак


| Угроза              | Меры                                                        |
| ------------------- | ----------------------------------------------------------- |
| Brute-force         | Rate limiting: 5 попыток / 15 мин на IP, 5 на email         |
| Credential stuffing | Та же rate limit + опционально CAPTCHA после 3 неудач       |
| Session hijacking   | HttpOnly cookie для refresh, Secure, SameSite=Strict (prod) |
| Token theft         | Короткий accessToken, rotation refresh                      |
| Replay              | JTI в JWT, проверка однократности при refresh               |


### 5.4. CORS и Cookie

**CORS для credentials:** при запросах с `credentials: 'include'` (cookie) сервер должен возвращать `Access-Control-Allow-Credentials: true` и `Access-Control-Allow-Origin` с **конкретным origin** (не `*`). Иначе cookie не будут отправляться в cross-origin запросах.

Только **refreshToken** передаётся через cookie:

- `HttpOnly` — всегда, недоступен из JS
- `Path=/api/v1/admin/auth` — cookie отправляются только на эндпоинты auth
- Имя cookie: `refreshToken` (или по конвенции проекта)
- `Domain` — из env `COOKIE_DOMAIN` (prod: `.example.com`; dev: пусто — для localhost)

**По окружению:**

- **Dev:** `Secure=false`, `SameSite=Lax` (для localhost с разными портами), `COOKIE_DOMAIN` пусто
- **Prod:** `Secure=true`, `SameSite=Strict`, `COOKIE_DOMAIN=.example.com` (из env)

**accessToken** передаётся в заголовке `Authorization: Bearer <token>` — клиент хранит в памяти и добавляет к каждому запросу.

### 5.5. Логирование и rate limiting

**Логи (pino и т.п.):**

- Успешные и неудачные попытки входа (без пароля)
- IP, User-Agent, timestamp, email (нормализованный)
- Не логировать токены и пароли

**Таблица `LoginAttempt` (БД):**

- Запись каждой попытки входа (успешной и неудачной)
- Используется **только для аудита** (анализ, отчётность)
- Поля: email, ipAddress, success, createdAt

**Real-time rate limiting:** выполняется отдельным механизмом (Throttler + Redis) — лимиты на IP для `/login`, `/register`, `/forgot-password`, `/reset-password` (5 попыток / 15 мин на IP). Не зависит от LoginAttempt.

## 6. Edge cases и обработка


| Сценарий                                       | Поведение                                                                   |
| ---------------------------------------------- | --------------------------------------------------------------------------- |
| Email уже зарегистрирован                      | 409 Conflict, `EMAIL_ALREADY_EXISTS`                                        |
| Неверный email или пароль                      | 401, `INVALID_CREDENTIALS` (одинаковый ответ — не раскрывать наличие email) |
| Аккаунт заблокирован                           | 403, `ACCOUNT_LOCKED`                                                       |
| Refresh token истёк                            | 401, `REFRESH_TOKEN_EXPIRED`                                                |
| Refresh token отозван                          | 401, `REFRESH_TOKEN_REVOKED`                                                |
| Refresh token использован повторно (reuse)     | Отзыв всей цепочки токенов пользователя (token family compromise)           |
| Access token истёк                             | 401, клиент вызывает `/refresh`                                             |
| Пустой/невалидный токен                        | 401, `UNAUTHORIZED`                                                         |
| Email не нормализован                          | `trim` + `lowercase` перед сохранением и при поиске                         |
| Email невалидный (формат, длина > 254)         | 400, валидация DTO                                                          |
| Регистрация с пустым именем                    | 400, валидация DTO                                                          |
| Регистрация EMAIL_PASSWORD без email           | 400, `EMAIL_REQUIRED` — для email+пароль email обязателен                   |
| Слишком слабый пароль                          | 400, `WEAK_PASSWORD`                                                        |
| Токен сброса пароля неверный/истёк/использован | 400, `INVALID_OR_EXPIRED_TOKEN`                                             |
| Logout без refresh cookie                      | 401, `UNAUTHORIZED`                                                         |
| Refresh без cookie                             | 401, `REFRESH_TOKEN_EXPIRED` (или `UNAUTHORIZED`)                           |


### 6.1. Блокировка аккаунта (опционально)

- После N неудачных попыток (например, 10) — блокировка на 30 мин
- Поле `AdminUser.lockedUntil` или отдельная таблица `AccountLock`

### 6.2. Очистка истёкших токенов

- Cron job: удаление записей `RefreshToken` где `expiresAt < now()` и `revokedAt IS NOT NULL`
- Рекомендуется хранить отозванные токены до истечения для детекции reuse
- Cron job для удаления истекших `PasswordResetToken`

## 7. Middleware и Guards

Для явного разделения по типам пользователей (Admin vs Client) guards имеют префикс контекста:

- **AdminAuthGuard:** проверка accessToken из заголовка `Authorization: Bearer <token>`, извлечение `adminUserId` в `request.user`. Принимает **только** access token: проверка `type === "access"` в JWT payload; при `type === "refresh"` или отсутствии type — 401.
- **AdminRefreshGuard:** для `/refresh` — проверка refreshToken из cookie. Принимает **только** refresh token: проверка `type === "refresh"` в JWT payload; при `type === "access"` или отсутствии type — 401.
- **Public:** декоратор для `/register`, `/login`, `/forgot-password`, `/reset-password` — обход AdminAuthGuard

**Проверка типа токена:** предотвращает перепутывание — если клиент по ошибке отправит access token на `/refresh` или refresh token на защищённый эндпоинт, guard вернёт 401.

В будущем Client API получит свои guards (например, `ClientAuthGuard` для проверки Telegram initData).

## 8. Конфигурация и зависимости

**Env-переменные:**

- `FRONTEND_URL` (или `APP_URL`) — базовый URL Web Admin для ссылки сброса пароля
- `COOKIE_DOMAIN` — домен для cookie refresh token (prod: `.example.com`; dev: пусто)
- `JWT_SECRET` — секрет для подписи JWT (access и refresh)
- `JWT_ACCESS_EXPIRES` — время жизни access token (например, `15m`)
- `JWT_REFRESH_EXPIRES` — время жизни refresh token (например, `7d`)

**Зависимости:**

- `@nestjs/jwt` — генерация и верификация JWT
- `argon2` или `bcrypt` — хеширование паролей
- `class-validator`, `class-transformer` — DTO
- Сервис отправки email (для forgot-password) — **заглушка** на данном этапе: логирует запрос (email, сгенерированная ссылка, token) для отладки; письмо не отправляется. Токен создаётся и сохраняется в БД — flow reset-password можно тестировать через логи.
