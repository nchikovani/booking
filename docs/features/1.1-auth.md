## 1. Обзор

Аутентификация и авторизация для Web Admin. Пользователи (владельцы бизнесов, администраторы) входят через учётные данные и получают доступ к Admin API на основе токенов.

### 1.1. Область реализации

| Область             | Применимо | Примечание                                      |
| ------------------- | --------- | ----------------------------------------------- |
| Backend (API)       | ☑         | NestJS, AdminAuthModule, эндпоинты, guards       |
| Frontend (Admin)    | ☐         | Web Admin — логин, формы (использует API)       |
| Frontend (Mini App) | ☐         | —                                               |
| База данных         | ☑         | Prisma, AdminUser, AuthProvider, RefreshToken и др. |

### 1.2. Контекст

В системе **два типа пользователей**, каждый со своей таблицей и потоком аутентификации. Данный документ описывает **Admin User**.

| Сущность     | Назначение                                                                  |
| ------------ | --------------------------------------------------------------------------- |
| **AdminUser**| Web Admin — владельцы бизнесов, администраторы. Email + пароль, токены      |
| **Client**   | Telegram Mini App — клиенты, записывающиеся на услуги (Feature 1.8)          |

## 3. Критерии приёмки

- [ ] Регистрация, вход, выход, refresh работают через Admin API
- [ ] Access token в Authorization, refresh token в HttpOnly cookie
- [ ] GET /me возвращает текущего пользователя и создаёт бизнес при первом входе
- [ ] Forgot/reset password: токен в письме, инвалидация после использования
- [ ] Rate limiting на login, register, forgot-password
- [ ] Guards: AdminAuthGuard, AdminRefreshGuard, проверка типа токена
- [ ] Cron: очистка истёкших RefreshToken и PasswordResetToken

## 4. Технический дизайн

### 4.1. Расширяемость способов входа

Реализован **только email + пароль**. Архитектура расширяема — добавление OAuth, magic link и т.д.:

```
┌─────────────────────────────────────────────────────────────┐
│                    AuthStrategy (интерфейс)                  │
├─────────────────────────────────────────────────────────────┤
│  + authenticate(credentials): Promise<AuthResult>           │
└─────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ EmailPassword   │  │ OAuthGoogle     │  │ MagicLink       │
│ Strategy        │  │ Strategy (future)│  │ (future)        │
└─────────────────┘  └─────────────────┘  └─────────────────┘
```

- **Strategy pattern** — каждый способ входа реализует общий интерфейс
- **AuthProvider** в БД — привязка учётной записи к способу входа
- Регистрация стратегий через DI (NestJS)

### 4.2. Токены

| Токен            | Назначение                     | Время жизни | Передача                                  |
| ---------------- | ------------------------------ | ----------- | ----------------------------------------- |
| **accessToken**  | Доступ к защищённым эндпоинтам | 15 мин      | Заголовок `Authorization: Bearer <token>` |
| **refreshToken** | Обновление accessToken         | 7 дней      | HttpOnly cookie (защита от XSS)           |

Клиент хранит accessToken в памяти и отправляет в `Authorization`. Refresh token — только в cookie. Для `/refresh` запросы отправляются с `credentials: 'include'`.

**Флоу при загрузке приложения:** accessToken теряется при перезагрузке. При загрузке: если есть refresh cookie, вызвать `/refresh` и получить новый accessToken. Если 401 — редирект на логин.

**Формат:** JWT для обоих токенов. **Refresh token rotation:** при каждом обновлении выдаётся новый refreshToken, старый инвалидируется.

### 4.3. Middleware и Guards

- **AdminAuthGuard:** проверка accessToken из заголовка `Authorization: Bearer <token>`, извлечение `adminUserId` в `request.user`. Принимает **только** access token: проверка `type === "access"` в JWT payload.
- **AdminRefreshGuard:** для `/refresh` — проверка refreshToken из cookie. Принимает **только** refresh token: проверка `type === "refresh"` в JWT payload.
- **Public:** декоратор для `/register`, `/login`, `/forgot-password`, `/reset-password`, `/refresh`, `/logout` — обход AdminAuthGuard (для `/refresh` и `/logout` используется AdminRefreshGuard)

## 5. Схема базы данных

### 5.1. Таблица `AdminUser`

```prisma
model AdminUser {
  id        String   @id @default(uuid())
  email     String?  @unique  // для EMAIL_PASSWORD — обязателен (валидация в DTO)
  firstName String?
  lastName  String?
  emailVerifiedAt DateTime?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  authProviders       AuthProvider[]
  refreshTokens        RefreshToken[]
  passwordResetTokens  PasswordResetToken[]
}
```

### 5.2. Таблица `AuthProvider`

```prisma
enum AuthProviderType {
  EMAIL_PASSWORD
}

model AuthProvider {
  id            String          @id @default(uuid())
  adminUserId   String          @map("admin_user_id")
  type          AuthProviderType
  providerId    String?         @map("provider_id")  // для OAuth: external id
  createdAt     DateTime        @default(now())

  adminUser     AdminUser       @relation(fields: [adminUserId], references: [id], onDelete: Cascade)
  credential    UserCredential?  // только для EMAIL_PASSWORD

  @@unique([adminUserId, type])
  @@map("auth_providers")
}
```

### 5.3. Таблица `UserCredential`

```prisma
model UserCredential {
  id             String   @id @default(uuid())
  authProviderId String   @unique @map("auth_provider_id")
  passwordHash   String   @map("password_hash")  // bcrypt/argon2
  updatedAt      DateTime @updatedAt

  authProvider   AuthProvider @relation(fields: [authProviderId], references: [id], onDelete: Cascade)

  @@map("user_credentials")
}
```

### 5.4. Таблица `RefreshToken`

```prisma
model RefreshToken {
  id           String    @id @default(uuid())
  adminUserId  String    @map("admin_user_id")
  tokenHash    String    @unique @map("token_hash")  // SHA-256 от токена
  deviceInfo   String?   @map("device_info")        // User-Agent, для аудита
  expiresAt    DateTime  @map("expires_at")
  revokedAt    DateTime? @map("revoked_at")
  replacedBy   String?   @map("replaced_by")        // id нового токена при rotation
  createdAt    DateTime  @default(now())

  adminUser    AdminUser @relation(fields: [adminUserId], references: [id], onDelete: Cascade)

  @@index([adminUserId])
  @@index([tokenHash])
  @@index([expiresAt])  // для очистки истёкших
  @@map("refresh_tokens")
}
```

### 5.5. Таблица `LoginAttempt`

```prisma
model LoginAttempt {
  id        String   @id @default(uuid())
  email     String   // нормализованный
  ipAddress String?  @map("ip_address")
  success   Boolean
  createdAt DateTime @default(now())

  @@index([email, createdAt])
  @@index([ipAddress, createdAt])
  @@map("login_attempts")
}
```

### 5.6. Таблица `PasswordResetToken`

```prisma
model PasswordResetToken {
  id        String   @id @default(uuid())
  adminUserId String @map("admin_user_id")
  tokenHash  String   @unique @map("token_hash")  // SHA-256 от токена
  expiresAt  DateTime @map("expires_at")
  usedAt     DateTime? @map("used_at")
  createdAt  DateTime @default(now())

  adminUser  AdminUser @relation(fields: [adminUserId], references: [id], onDelete: Cascade)

  @@index([adminUserId])
  @@index([tokenHash])
  @@index([expiresAt])
  @@map("password_reset_tokens")
}
```

## 6. API эндпоинты

Base path: `/api/v1/admin/auth`

| Метод | Путь               | Описание                           | Доступ                       |
| ----- | ------------------ | ---------------------------------- | ---------------------------- |
| POST  | `/register`        | Регистрация (email + пароль)       | Public                       |
| POST  | `/login`           | Вход (email + пароль)              | Public                       |
| POST  | `/refresh`         | Обновление accessToken             | Refresh cookie               |
| POST  | `/logout`          | Выход (инвалидация текущей сессии) | Refresh cookie (обязательно) |
| GET   | `/me`              | Текущий пользователь               | Access token (Authorization) |
| POST  | `/forgot-password` | Запрос сброса пароля               | Public                       |
| POST  | `/reset-password`  | Сброс пароля по токену             | Public                       |

### 6.1. Регистрация

**POST** `/api/v1/admin/auth/register`

Создание AdminUser, AuthProvider, UserCredential, RefreshToken — **в одной транзакции** (Prisma `$transaction`).

**Request body:**
```json
{
  "email": "user@example.com",
  "password": "SecureP@ssw0rd",
  "firstName": "Иван",
  "lastName": "Иванов"
}
```

**Response 201:** refreshToken в Set-Cookie (HttpOnly), accessToken и user в body
```json
{
  "status": "success",
  "data": {
    "user": {
      "id": "uuid",
      "email": "user@example.com",
      "firstName": "Иван",
      "lastName": "Иванов"
    },
    "accessToken": "eyJ...",
    "expiresIn": 900
  }
}
```

**Errors:** `400 VALIDATION_FAILED` | `409 EMAIL_ALREADY_EXISTS` | `429 RATE_LIMIT_EXCEEDED`

| Сценарий | Поведение |
| -------- | --------- |
| Email уже зарегистрирован | 409 Conflict, `EMAIL_ALREADY_EXISTS` |
| Превышен лимит попыток | 429, `RATE_LIMIT_EXCEEDED` |

### 6.2. Вход

**POST** `/api/v1/admin/auth/login`

**Request body:**
```json
{
  "email": "user@example.com",
  "password": "SecureP@ssw0rd"
}
```

**Response 200:** refreshToken в Set-Cookie (HttpOnly), accessToken и user в body
```json
{
  "status": "success",
  "data": {
    "user": { "id", "email", "firstName", "lastName" },
    "accessToken": "eyJ...",
    "expiresIn": 900
  }
}
```

**Errors:** `400 VALIDATION_FAILED` | `401 INVALID_CREDENTIALS` | `403 ACCOUNT_LOCKED` | `429 RATE_LIMIT_EXCEEDED`

| Сценарий | Поведение |
| -------- | --------- |
| Неверный email или пароль | 401, `INVALID_CREDENTIALS` |
| Аккаунт заблокирован | 403, `ACCOUNT_LOCKED` |

### 6.3. Обновление токена

**POST** `/api/v1/admin/auth/refresh`

- Cookie: `refreshToken` (Path: `/api/v1/admin/auth`)
- Response: тот же формат, что и при login (refreshToken в Set-Cookie, accessToken в body)

**Errors:** `401 REFRESH_TOKEN_EXPIRED` | `401 REFRESH_TOKEN_REVOKED` | `401 UNAUTHORIZED`

### 6.4. Выход

**POST** `/api/v1/admin/auth/logout`

- Cookie: `refreshToken` (обязательно) — инвалидирует текущую сессию
- Response: `{ "status": "success" }` + `Set-Cookie: refreshToken=; Max-Age=0; Path=/api/v1/admin/auth`
- При заданном COOKIE_DOMAIN — Domain=COOKIE_DOMAIN

**Errors:** `401 UNAUTHORIZED` (если cookie отсутствует)

### 6.5. Текущий пользователь (GET /me)

**GET** `/api/v1/admin/auth/me`

Возвращает данные текущего AdminUser. **При первом вызове** (если у пользователя ещё нет бизнеса) — автоматически создаётся Business и BusinessMember (роль OWNER). См. [Feature 1.2](1.2-business.md).

**Response 200:**
```json
{
  "status": "success",
  "data": { "id", "email", "firstName", "lastName" }
}
```

**Порядок вызовов при загрузке Web Admin:** `GET /me` вызывается первым — создаёт бизнес при первом входе и возвращает user. Далее могут выполняться запросы `GET /businesses`, `GET /businesses/:id`.

### 6.6. Забытый пароль

**POST** `/api/v1/admin/auth/forgot-password`

**Request body:**
```json
{
  "email": "user@example.com"
}
```

**Response 200** (всегда — не раскрывать наличие email):
```json
{
  "status": "success",
  "data": { "message": "Если аккаунт существует, на email отправлена ссылка для сброса пароля" }
}
```

Токен создаётся **только если** найден AdminUser с AuthProvider типа `EMAIL_PASSWORD`. Ссылка вида `{FRONTEND_URL}/reset-password?token=...`. Токен действителен 1 час. При новом запросе для того же email — удалить все активные `PasswordResetToken` этого пользователя.

### 6.7. Сброс пароля

**POST** `/api/v1/admin/auth/reset-password`

**Request body:**
```json
{
  "token": "одноразовый токен из ссылки",
  "password": "NewSecureP@ssw0rd"
}
```

**Response 200:**
```json
{
  "status": "success",
  "data": { "message": "Пароль успешно изменён" }
}
```

После успешного сброса: токен инвалидируется (`usedAt`), **все refresh token пользователя** отзываются.

**Errors:** `400 INVALID_OR_EXPIRED_TOKEN`

## 9. Безопасность и доступ

### 9.1. Пароли

- **Хеширование:** Argon2id (или bcrypt) с солью
- **Минимальные требования:** 8+ символов, буквы + цифры, опционально спецсимволы

### 9.2. Токены

- **AccessToken (JWT):** payload `{ sub: adminUserId, iat, exp, type: "access" }`
- **RefreshToken:** JWT с `type: "refresh"`, проверка в БД по `tokenHash`
- **Refresh rotation:** при каждом `/refresh` — новый refresh, старый помечается `revokedAt`, `replacedBy`

### 9.3. Защита от атак

| Угроза              | Меры                                                        |
| ------------------- | ----------------------------------------------------------- |
| Brute-force         | Rate limiting: 5 попыток / 15 мин на IP, 5 на email         |
| Credential stuffing | Та же rate limit + опционально CAPTCHA после 3 неудач       |
| Session hijacking   | HttpOnly cookie для refresh, Secure, SameSite=Strict (prod) |
| Token theft         | Короткий accessToken, rotation refresh                      |
| Replay              | JTI в JWT, проверка однократности при refresh               |

### 9.4. CORS и Cookie

**CORS для credentials:** при `credentials: 'include'` сервер возвращает `Access-Control-Allow-Credentials: true`. В production `Access-Control-Allow-Origin` — конкретный origin (не `*`).

Только **refreshToken** передаётся через cookie:

- `HttpOnly` — всегда, недоступен из JS
- `Path=/api/v1/admin/auth` — cookie отправляются только на эндпоинты auth
- Имя cookie: `refreshToken`
- `Domain` — из env `COOKIE_DOMAIN` (prod: `.example.com`; dev: пусто)

**По окружению:**

- **Dev:** `Secure=false`, `SameSite=Lax`, `COOKIE_DOMAIN` пусто
- **Prod:** `Secure=true`, `SameSite=Strict`, `COOKIE_DOMAIN=.example.com`

**accessToken** передаётся в заголовке `Authorization: Bearer <token>`.

### 9.5. Логирование и rate limiting

**Логи (pino):** успешные и неудачные попытки входа (без пароля); IP, User-Agent, timestamp, email (нормализованный). Не логировать токены и пароли.

**Таблица `LoginAttempt`:** запись каждой попытки входа — для аудита.

**Real-time rate limiting:**

- **По IP:** Throttler (in-memory) — 5 попыток / 15 мин на IP
- **По email:** Redis — 5 попыток / 15 мин на email (для `/login`, `/register`, `/forgot-password`)

## 11. Конфигурация и зависимости

**Env-переменные:**

- `FRONTEND_URL` (или `APP_URL`) — базовый URL Web Admin для ссылки сброса пароля
- `COOKIE_DOMAIN` — домен для cookie refresh token (prod: `.example.com`; dev: пусто)
- `JWT_SECRET` — секрет для подписи JWT (access и refresh)
- `JWT_ACCESS_EXPIRES` — время жизни access token (например, `15m`)
- `JWT_REFRESH_EXPIRES` — время жизни refresh token (например, `7d`)

**Зависимости:**

- `@nestjs/jwt` — генерация и верификация JWT
- `argon2` или `bcrypt` — хеширование паролей
- `class-validator`, `class-transformer` — DTO
- Сервис отправки email (forgot-password): заглушка — логирует запрос, письмо не отправляется
