# Потенциальные проблемы системы записи и их решения

Документ описывает типичные проблемы, с которыми сталкиваются системы онлайн-записи, и оптимальные подходы к их решению. Предназначен для планирования и проектирования функциональности.

---

## 1. Конфликты и гонки

### 1.1 Двойная запись (race condition)

**Проблема:** Два клиента одновременно выбирают один и тот же слот и оба нажимают «Подтвердить». Оба получают запись на одно время.

**Решение:**
- Оптимистичная блокировка: при подтверждении записи проверять, что слот всё ещё свободен.
- Использовать транзакцию с `SELECT ... FOR UPDATE` или аналогичный механизм.
- Если слот занят — вернуть ошибку «Время уже занято», обновить список слотов и предложить выбрать другое.

### 1.2 Админ создаёт запись вручную, пока клиент бронирует

**Проблема:** Клиент в Mini App выбирает слот, админ в панели создаёт запись на то же время. Возможна двойная запись.

**Решение:**
- Единая проверка при сохранении записи (из Mini App или админки).
- Опционально: временный «холд» слота на 30–60 секунд при начале оформления (сложнее, но улучшает UX).

### 1.3 Расписание изменилось во время записи

**Проблема:** Клиент выбрал слот, пока заполнял форму — админ добавил исключение (отпуск) на этот день. Клиент подтверждает — слот уже недоступен.

**Решение:**
- При финальном подтверждении всегда перепроверять доступность слота.
- Если слот занят или недоступен — показать сообщение «Время больше недоступно, выберите другое» и обновить список.

## 2. Время и длительность

### 2.1 Часовой пояс

**Проблема:** Клиент в другом часовом поясе, бизнес в Москве. В какое время показывать слоты?

**Решение:**
- Хранить все даты и время в UTC в базе данных.
- В настройках компании — часовой пояс (например, Europe/Moscow).
- Показывать клиенту время в часовом поясе компании (или клиента, если продукт международный).

### 2.2 Переход на летнее/зимнее время

**Проблема:** Дважды в год время сдвигается на час. Фиксированный offset может сломаться.

**Решение:**
- Использовать IANA timezone (Europe/Moscow и т.п.), не фиксированный offset.
- Библиотеки: `date-fns-tz`, `luxon`, `dayjs` с плагином timezone.

### 2.3 Технический перерыв после услуги

**Проблема:** Услуга 60 мин + 15 мин перерыв. Следующий клиент не может начать в 10:01 — только в 10:15 (или 10:00 + 60 + 15 = 11:15).

**Решение:**
- Поле «Технический перерыв после записи (мин)» в настройках услуги.
- При генерации слотов: `nextSlotStart = previousSlotEnd + serviceDuration + buffer`.

### 2.4 Последний слот дня

**Проблема:** Работа до 18:00, услуга 60 мин. Последний слот должен быть 17:00, а не 17:30.

**Решение:**
- Генерировать слоты с условием: `slotStart + slotDuration <= endOfWork`.
- Не включать слоты, которые «вылезают» за конец рабочего дня.

## 2.5 Минимальное время до записи

**Проблема:** Клиент записывается на «через 30 минут», но администратор не успевает подготовиться.

**Решение:**
- Настройка «Минимальный срок до записи»: 0, 1, 2, 4, 24 часа.
- Не показывать слоты раньше `now + minNotice`.

## 3. Отмена и неявка

### 3.1 Политика отмены

**Проблема:** Клиент отменяет за 1 час до визита — слот потерян, другой клиент не успел записаться.

**Решение:**
- Настройка «Бесплатная отмена до N часов до визита».
- При поздней отмене — уведомление администратору (опционально: «штраф» при повторных нарушениях).
- Текст в условиях записи: «Отмена возможна не позднее чем за 24 часа».

### 3.2 Неявка (no-show)

**Проблема:** Клиент не пришёл, не отменил. Слот потерян.

**Решение:**
- Статус записи «Неявка»; администратор отмечает вручную.
- Опционально: для клиентов с повторными неявками — требовать предоплату или ограничить возможность записи.

## 4. Изменение данных после записи

### 4.1 Сотрудник уволен / убран

**Проблема:** У сотрудника есть записи на будущее. Его удалили из системы.

**Решение:**
- При деактивации/удалении: предупреждение «У сотрудника N записей на будущее».
- Варианты: массовая отмена с уведомлением клиентам; переназначение на другого сотрудника (если возможно).
- Или: блокировать удаление, пока есть записи; перенести в «неактивный» и обработать вручную.

### 4.2 Сотрудник удалён из услуги

**Проблема:** Услуга «Стрижка» привязана к Анне. Админ убирает Анну из услуги. У Анны есть записи на «Стрижку».

**Решение:**
- При отвязке: проверка «У сотрудника N записей на эту услугу».
- Варианты: запретить отвязку; или отменить/переназначить записи с уведомлением.

### 4.3 Услуга удалена

**Проблема:** Услуга «Маникюр» удалена, но есть записи на неё.

**Решение:**
- Мягкое удаление: услуга помечается как «удалена», не показывается в каталоге, но записи остаются.
- Или: блокировать удаление при наличии будущих записей; предложить «скрыть» вместо удаления.

### 4.4 Изменение цены услуги

**Проблема:** Услуга стоила 1000₽, подняли до 1500₽. Старые записи — по какой цене?

**Решение:**
- Существующие записи хранят цену на момент записи.
- Новые записи — по актуальной цене.
- В карточке записи — показывать цену из записи, не из услуги.

### 4.5 Изменение длительности услуги

**Проблема:** Услуга была 60 мин, изменили на 90 мин. Существующие записи — конфликты?

**Решение:**
- Существующие записи хранят длительность на момент записи.
- При изменении длительности — проверка: «Есть записи в этот день. Возможны конфликты. Продолжить?»
- При конфликте — предупреждение или запрет изменения.

## 5. Расписание и исключения

### 5.1 Выходные компании

**Проблема:** Праздник, ремонт — вся компания не работает. Сейчас только исключения по сотрудникам.

**Решение:**
- Отдельная сущность «Нерабочие дни компании» (глобальные).
- При генерации слотов: `companyWorkDays AND employeeWorkDays AND NOT companyExceptions AND NOT employeeExceptions`.

### 5.2 Пересечение исключений

**Проблема:** На один день: «отпуск» (недоступность) и «особый график 10:00–16:00». Что приоритетнее?

**Решение:**
- Чёткое правило: «Недоступность» перекрывает «Особый график».
- Если сотрудник в отпуске — он недоступен, даже если есть особый график.
- Порядок применения: базовый → особый график → недоступность (перезаписывает).

### 5.3 Исключение на дату с существующими записями

**Проблема:** Админ добавляет отпуск 15–22 марта. На эти дни уже есть 5 записей.

**Решение:**
- При добавлении исключения: проверка «На этот период есть N записей у сотрудника».
- Предупреждение: «Записи будут отменены или потребуют переназначения. Продолжить?»
- Опционально: массовая отмена с уведомлением клиентам.

### 5.4 Цикличный график (2/2, 3/3)

**Проблема:** График «2 рабочих, 2 выходных» — сдвиг при смене даты начала цикла.

**Решение:**
- Хранить: `cycleStartDate`, `workDays`, `restDays`.
- Формула: `dayIndex = (date - cycleStartDate).days % (workDays + restDays)`.
- `dayIndex < workDays` → рабочий день.

## 6. Уведомления

### 6.1 Запись за 20 часов до визита

**Проблема:** Напоминание «за 24 часа» уже не сработает — до визита меньше суток.

**Решение:**
- Условная логика: если `hoursUntil < 24` — не отправлять «за 24ч», отправить напоминание «за 2ч» в нужное время.
- Или: при создании записи планировать только те напоминания, которые ещё актуальны.

### 6.2 Клиент не в Telegram

**Проблема:** Mini App открыт в браузере, но уведомления идут в Telegram-бота. Клиент не получит.

**Решение:**
- Требовать открытие Mini App через Telegram (TMA) — тогда клиент в экосистеме.
- Или: fallback — SMS (если есть телефон), email.
- В условиях: «Для уведомлений необходим Telegram».

### 6.3 Спам уведомлениями

**Проблема:** Слишком много уведомлений — клиент отписывается.

**Решение:**
- Оптимально: подтверждение + 1–2 напоминания (24ч, 2ч).
- Настройки в админке: «Какие уведомления отправлять» (чекбоксы).
- В боте — не дублировать то, что уже в Mini App.

### 6.4 Запись отменена после отправки напоминания

**Проблема:** Отправили «за 24ч», клиент отменил. Следующее напоминание «за 2ч» не должно уйти.

**Решение:**
- Перед отправкой любого уведомления проверять статус записи.
- Отменённые записи — не отправлять уведомления.

## 7. Бизнес-правила

### 7.1 Один клиент — много записей

**Проблема:** Клиент записывается на 10 слотов вперёд и не приходит.

**Решение:**
- Опционально: лимит «Максимум N активных записей на клиента» (например, 3).
- При превышении — «У вас уже есть N записей. Дождитесь визита или отмените».

### 7.2 Повторные неявки

**Проблема:** Клиент 3 раза не пришёл. Слоты теряются.

**Решение:**
- Флаг «часто неявки» в профиле клиента.
- Опции: требовать предоплату; ограничить возможность записи; предупреждение при записи.

### 7.3 Запись «на любого» мастера

**Проблема:** Клиент выбирает «любой свободный» — не привязан к конкретному сотруднику.

**Решение:**
- Агрегировать слоты от всех сотрудников, умеющих услугу.
- При записи — назначать свободному (или первый приоритетный).
- При конфликте — перепроверять доступность выбранного слота.

### 7.4 Не все сотрудники делают услугу

**Проблема:** Услуга привязана к 2 из 5 сотрудников. Показывать слоты только этих двух.

**Решение:**
- Генерировать слоты только по сотрудникам, привязанным к услуге.
- В настройках услуги — список сотрудников с индивидуальной ценой/длительностью.

## 8. Платёж и предоплата

### 8.1 Предоплата для подтверждения

**Проблема:** Без предоплаты — много неявок.

**Решение:**
- Интеграция с платёжной системой (ЮKassa, Stripe и т.п.).
- Запись подтверждается только после оплаты.

### 8.2 Возврат при отмене

**Проблема:** Клиент отменил — что с предоплатой?

**Решение:**
- Правила: «Полный возврат при отмене за 24ч», «Частичный за 12ч», «Без возврата за 2ч».
- Настройки в админке: политика возврата.

### 8.3 Оплата при визите

**Проблема:** Оплата в салоне, не онлайн.

**Решение:**
- Статус записи «Оплачено» — администратор отмечает вручную.
- Отчётность: оплаченные vs неоплаченные записи.

## 9. UX и граничные случаи

### 9.1 Нет свободных слотов

**Проблема:** На выбранную дату все слоты заняты.

**Решение:**
- Сообщение: «На эту дату нет свободного времени».
- Предложить: другую дату, другого сотрудника.

### 9.2 Нет сотрудников на услугу

**Проблема:** Услуга без привязки к сотрудникам — слоты не генерируются.

**Решение:**
- Не показывать услугу в записи (или с пометкой «Запись через администратора»).
- При создании услуги — предупреждение «Добавьте сотрудников для онлайн-записи».

### 9.3 Пустой календарь

**Проблема:** Нет записей — пустой экран.

**Решение:**
- Empty state: «Пока нет записей» + призыв к действию (создать первую запись, настроить расписание).

### 9.4 Клиент без Telegram

**Проблема:** Mini App открыт вне Telegram (если технически возможно).

**Решение:**
- Ограничить: Mini App только через TMA (Telegram Mini App).
- Или: без авторизации — только просмотр; для записи — обязательна привязка к Telegram.

## 10. Данные и безопасность

### 10.1 Согласие на обработку ПД

**Проблема:** Юридические требования к обработке персональных данных.

**Решение:**
- Чекбокс при записи: «Согласен на обработку персональных данных».
- Хранить: дату/время согласия, IP (опционально).
- Ссылка на политику конфиденциальности.

### 10.2 Хранение истории

**Проблема:** Как долго хранить записи? GDPR — право на удаление.

**Решение:**
- Политика: «История записей хранится N лет» (настраиваемо).
- Функция экспорта данных клиента.
- Возможность удаления по запросу (с учётом бухгалтерии, если есть).

### 10.3 Доступ к чужим записям

**Проблема:** Клиент видит чужие записи (баг, подбор ID).

**Решение:**
- Строгая проверка: записи только по `telegramUserId` (или другому идентификатору клиента).
- API: фильтр по текущему пользователю, не передавать данные других.

## 11. Производительность

### 11.1 Генерация слотов

**Проблема:** Расчёт слотов на 30 дней для 10 сотрудников × 20 услуг — тяжёлая операция.

**Решение:**
- Генерировать слоты по запросу (при выборе даты), не заранее.
- Кэширование на 1–5 минут (с учётом частоты изменений).
- Ограничение горизонта (30–90 дней).

### 11.2 Много исключений

**Проблема:** У сотрудника сотни исключений — медленные запросы.

**Решение:**
- Индексы по `staffId`, `dateFrom`, `dateTo`.
- Запросы только для нужного диапазона дат.
- Пагинация при отображении списка исключений.

## 12. Локализация и формат

### 12.1 Формат даты

**Проблема:** 18.03.2025 vs 2025-03-18 vs March 18, 2025.

**Решение:**
- Использовать локаль (ru-RU) для отображения.
- В настройках — формат даты (если продукт мультиязычный).

### 12.2 День недели

**Проблема:** Пн vs Mon vs Понедельник.

**Решение:**
- Локаль для отображения.
- В админке — короткие названия (Пн, Вт, …) для компактности.

### 12.3 Первый день недели

**Проблема:** Понедельник (Европа) vs воскресенье (США).

**Решение:**
- Настройка в локали или в настройках компании.
- По умолчанию для ru — понедельник.

## Приоритеты для MVP

**Обязательно:**
- Двойная запись (проверка при подтверждении)
- Технический перерыв и последний слот дня
- Горизонт записи (максимальный срок)
- Часовой пояс (хотя бы один для компании)

**Желательно:**
- Предупреждение при исключении на дату с записями
- Минимальное время до записи
- Политика отмены (хотя бы текст в условиях)

- Предоплата
- Лимиты на клиентов
- Обработка увольнения сотрудника
